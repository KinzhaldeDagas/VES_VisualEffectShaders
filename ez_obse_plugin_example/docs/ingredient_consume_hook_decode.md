## Observed Oblivion Flow

### `Actor_EquipItem` dispatch

`Actor_EquipItem` is at `0x005FAEA0`.

The inventory equip switch has an Ingredient case at `0x005FB2BD`. `kFormType_Ingredient` is enum value `25` in xOBSE `GameForms.h`.

Relevant instructions:

```asm
005FB2BD  mov     ebx, [esp+2Ch+arg_8]        ; case 25 / Ingredient
005FB2C1  push    2
005FB2C3  push    ebx
005FB2C4  push    esi
005FB2C5  call    Script_AddEventToExtraScript
005FB2D4  call    eax                         ; item vtbl +0x78, quest-item check
005FB2DA  cmp     esi, ds:0B333C4h            ; PlayerRef
005FB2E0  setnz   cl
005FB2E3  push    ecx                         ; bool recursive/success gate
```

For the player, the first `Actor_EquipIngredient?` call receives `arg2=false`; for non-player actors it receives `arg2=true`.

### `Actor_EquipIngredient?`

Function start: `0x005E44C0`.

IDA decompiles the signature as:

```cpp
void __userpurge Actor_EquipIngredient?(
    PlayerCharacter* actor@<ecx>,
    TESForm* item,
    BaseExtraList* extraList,
    char recursiveOrConfirmed);
```

Observed behavior:

- Null item exits.
- Quest item exits after queuing the "no eat quest item" message.
- If not in the save/load guard, it marks equip script events:

```asm
005E4511  push    2
005E4513  push    ecx                         ; item extra list
005E4514  push    esi                         ; actor
005E4515  call    Script_AddEventToExtraScript
005E451A  push    2
005E451C  lea     edx, [esi+44h]              ; actor base extra list
005E451F  push    edx
005E4520  push    edi                         ; ingredient
005E4521  call    Script_AddEventToExtraScript
```

- If `actor == PlayerRef && recursiveOrConfirmed == false`, it increments player misc stat `0xB`, calls `Player_EquipPotion?`, and removes one item only if that function returns true.
- Otherwise it sets the active magic item, uses it, removes one item for non-player actors, and refreshes the player path through `sub_5C1900`.

### `Player_EquipPotion?`

Function start: `0x006665E0`.

Despite the name, this handles both potions and ingredients on the player success path.

Relevant instructions:

```asm
006665E3  mov     edi, [esp+0Ch+arg_0]        ; MagicItem*
006665F2  call    edx                         ; magic item type
006665F4  cmp     eax, 7                      ; AlchemyItem
006665F9  jnz     short loc_66662D            ; non-potion path, includes Ingredient
...
0066662D  push    1
0066662F  push    edi
00666630  call    sub_5E45F0
00666635  push    edi
00666636  lea     ecx, [esi+68h]
00666639  call    MagicTarget_ProcessEffectsFromItem
0066663E  pop     edi
0066663F  pop     esi
00666640  mov     al, bl                      ; success true
```

`sub_5E45F0` then checks magic item type:

- type `7`: `AlchemyItem`, calls `Actor_ConsumePotion?`
- type `8`: `IngredientItem`, calls `Actor_EquipIngredient?` with `recursiveOrConfirmed=true`

This is why the xOBSE ingredient hook appears to ignore the first player call in this decoded path and fires only on the second observed player-success call.

## Existing xOBSE Hook

xOBSE already installs an ingredient event hook:

```cpp
static const UInt32 s_hookAddr = 0x005E4515;
WriteRelCall(s_hookAddr, (UInt32)&OnEatIngredientHook);
```

At the replaced call site:

- `ESI` is the eating `Actor*`.
- `EDI` is the `IngredientItem*`.
- `[esp+0x24]` is `Actor_EquipIngredient?`'s `recursiveOrConfirmed` boolean.

xOBSE checks that boolean before dispatching `OnEatIngredient`, then jumps to the original `Script_AddEventToExtraScript` call.

Do not overwrite `0x005E4515` from `MoonSugarEffect`; that risks clobbering xOBSE's `OnEatIngredient` event hook.

Current status: `0x005E4515` is archived as `StaleDoNotPatch_OnEatIngredientConflictSite`. It is retained here only to document the xOBSE conflict and must not be used as active VES hook guidance.

## Current Moon Sugar Trigger Point

For the current first-person drug distortion effect, the observed player-success path is enough. The current lowest-conflict native hook is the `Player_EquipPotion?` call to `MagicTarget_ProcessEffectsFromItem` at `0x00666639`.

Why this hook:

- It is after `Player_EquipPotion?` has passed the potion-cap success gate.
- It sees both player potion and player ingredient consumption.
- `EDI` and the stack argument both still hold the consumed `MagicItem*`.
- It avoids overwriting xOBSE's `0x005E4515` hook.
- It naturally excludes NPC eating, which cannot directly need a screen-space drug shader.

Hook shape:

```cpp
using ProcessEffectsFromItem = void (__thiscall*)(void* magicTarget, MagicItem* item);
static ProcessEffectsFromItem OriginalProcessEffectsFromItem =
    reinterpret_cast<ProcessEffectsFromItem>(0x006A23B0);

void __fastcall ProcessEffectsFromItemHook(void* magicTarget, UInt32, MagicItem* item)
{
    OriginalProcessEffectsFromItem(magicTarget, item);

    if (item && GetMagicItemType(item) == 8 && IsMoonSugar(item)) {
        StartVESDistortion();
    }
}
```

The direct call target for `MagicTarget_ProcessEffectsFromItem` is `0x006A23B0`; the player-success call site is `0x00666639`.

Implemented in `main.cpp`:

- Replaces the call at `0x00666639` with `ProcessEffectsFromItemHook`.
- Validates the original five bytes at `0x00666639` before patching. For Oblivion 1.2.0.416 the expected instruction bytes are `E8 72 BD 03 00`, the decoded direct call to `0x006A23B0`.
- Calls the original `MagicTarget_ProcessEffectsFromItem` first.
- Checks the engine magic-item type through vtable slot `+0x18`; type `8` is the Ingredient path observed in Oblivion IDA.
- Recovers the owning `TESForm*` by subtracting `0x24` from the `MagicItem*`, matching the `MagicItemObject::magicItem` layout in xOBSE.
- Matches Moon Sugar using embedded DLL defaults: Morroblivion Moon Sugar local form ID `000A0028`, load-order byte ignored.
- Starts native effect state and exports `VES_IsActive`, `VES_GetIntensity`, and `VES_GetRemainingMilliseconds` for the shader/rendering side.
- Tracks remaining time with signed `GetTickCount()` deltas rather than direct integer ordering, so short Moon Sugar effects remain correct across the Windows tick counter wrap.
- Settings that previously lived in `Data\OBSE\Plugins\MoonSugarEffect.ini` are embedded directly in the DLL. `VESReloadSettings` now resets those embedded defaults and releases shader resources; it does not read an external INI.
- Consumed Moon Sugar stacks duration up to the embedded stack cap; diagnostic `VESDistortion` starts the same MoonSugar75 visual state for repeatable bring-up tests.
- Expiration and explicit stop both reset the full native effect state, including dose count.
- Treats OBSE commands as diagnostic bring-up surface. The native consume hook is installed first; command registration failures are logged without unloading after process memory has been patched.
- Hooks Oblivion's main image-space shader call in `NiRenderer_Render` at `0x0040CF6E`. IDA shows this call invokes `ProcessImageSpaceShader(NiDX9Renderer*, BSRenderedTexture*, BSRenderedTexture*)` at `0x007B48E0`, after the scene render path has prepared the source and destination image-space textures.
- Validates the original five bytes at `0x0040CF6E` before patching: `E8 6D 79 3A 00`, the decoded direct call to `0x007B48E0`.
- Decoding `ProcessImageSpaceShader` shows it is a thin wrapper around `ImageSpaceShaderList::ProcessImageSpaceShader` at `0x008037D0`, using global image shader list `0x00B42D7C`. Earlier notes sometimes used `ImageShaderList`; keep that only as a historical alias.

- The Moon Sugar pass runs after the hooked engine image-space call returns, so it operates on the observed image-space result for that call path instead of competing with individual engine shaders. ## Renderer Decode: Object-Motion Limits

This pass focused on what Oblivion's renderer actually exposes if the goal is a Moon Sugar effect that feels attached to NIF objects rather than painted over the screen.

Confirmed IDA points:

- `NiRenderer_Render` begins at `0x0040C830`. In the normal gameplay caller at `0x0040D658`, Oblivion calls `NiRenderer_Render(..., 0)` for the world pass, then continues into later debug/interface/cursor work such as `sub_7B8400` and `sub_579260`/`MiscPass`. The Moon Sugar image-space hook is therefore inside the full rendered world scene pass, before later interface pass work.
- When `ImageSpaceEffectEnabled` is nonzero, `NiRenderer_Render` allocates the image-space source at `0x0040CA24-0x0040CA30` by calling `BSTextureManager_GetDefaultRenderTarget(g_textureManager, g_Renderer, 4)` and storing the returned `BSRenderedTexture*` in `var_38`.
- Default render-target type `4` is full default-render-target size. `GetRenderedSurfaceParameters` sets width/height from the renderer default RT group, usage to `0x22` normally or `0xA2` with `iMultisample >= 2`, and format to `0x71` when `UseHDR` is enabled, otherwise `0`.
- At `0x0040CF5F-0x0040CF6E`, the engine calls `ProcessImageSpaceShader(renderer, var_38, a2)`. In this call, `var_38` is the full-frame scene color source and `a2` is the destination/output target. If multisampling is active, the preceding path at `0x0040CF0E-0x0040CF5D` uses the D3D device `StretchRect` vtable slot (`+0x88`) to resolve/copy the source before image-space processing.
- After image-space processing, `var_38` is returned to the texture manager at `0x0040D04D-0x0040D054`.
- `BSImageSpaceShader::RenderShader` at `0x008028E0` is a full-screen image-space draw helper. It starts either the default target or the supplied destination `BSRenderedTexture`, stores the supplied source `BSRenderedTexture` at shader offset `+0x7C`, calls `SetupScreenSpaceCamera(0)`, renders a `NiScreenElements` quad through `sub_709C60`, then pops the render-target stack through `0x007D7110`.
- `sub_7AF5B0` is a related viewport-aware image-space render helper. It computes default-target/source texture ratios, writes ratio fields on the shader, binds the source, starts the supplied/default destination with `kClear_BACKBUFFER`, calls `SetupScreenSpaceCamera(viewport)`, draws the same screen element, then pops the render-target stack. This confirms TES4 image-space passes may adjust viewport/texture ratios but still draw a quad.
- `sub_7B4900` is only a thin wrapper over `sub_803570(imageSpaceShaderList, shader, renderer, source, destFlag)`. It is used by menu, water, canopy, and `MiscPass` paths to apply one shader through the global image-space quad. It is not independent of `imageSpaceShaderList` lifetime.
- `ImageSpaceShaderList::AssignScreenElements` at `0x008033D0` creates one `NiScreenElements` surface with four vertices and UVs `0..1`. This confirms the native image-space path is quad-based, not per-object.
- `BSTextureManager_GetDefaultRenderTarget` at `0x007C23C0` asks `GetRenderedSurfaceParameters` at `0x007C0D10` for target dimensions/format/usage, then allocates or reuses a `BSRenderedTexture` through `NiRenderer_GetRenderedSurface` at `0x007C1B50`.
- `NiRenderer_GetRenderedSurface` searches the texture manager's available rendered-surface list for a matching width/height/format/usage/depth signature. If it finds one, it marks it in use and moves it to the in-use list; otherwise it allocates a new `BSRenderedTexture` wrapper and moves that to the in-use list.
- `sub_7C1EE0` returns a `BSRenderedTexture` from the in-use list to the available list. Native image-space code consistently returns temporary targets through this manager path instead of raw releasing the render target immediately.
- Default render-target type `3` is full backbuffer size, usage `0x26`, no extra depth/stencil format. This is the temporary target used by active engine image-space shaders.
- `menuRenderedTexture` is a separate offscreen path. `sub_40D160` requests default render-target type `5`, calls `NiRenderer_Render(this, menuRenderedTexture)`, and later paths can apply a single image-space shader through `sub_7B4900`. That is useful contrast, but it is not the normal Moon Sugar world-render hook.
- `StartUsingRenderTarget` at `0x007D70A0` and pop helper `0x007D7110` maintain the native render-target stack at `0x00B45D74`/`0x00B45D78`, capped at ten entries. `StartUsingRenderTarget` ends the current target when needed, begins the requested/default target group through `sub_7D6A80`, then stores an incremented smart pointer in the stack. `0x007D7110` ends the current target, releases the top entry through `sub_7D7030`, then resumes the previous/default target with `kClear_NONE`.
- Scene-end helper `sub_7D7150` drains all render-target stack entries and ends any active target group before `NiRenderer_EndScene`/display. A plugin-owned render target must therefore either enter through these helpers and preserve stack balance or use a fully state-blocked D3D path that does not perturb `IsReady`, locks, current target group, or the stack.
- `NiDX9Renderer::BeginUsingRenderTargetGroup` at `0x007647E0` clears render targets `1..3`, binds color targets through `NiDX92DBufferData::SetRenderTarget` at `0x0076D3D0`, then binds a depth-stencil renderer-data object if the target group has one. If there is no depth object, it falls back to the renderer's current/default depth setup.
- `NiDX9DepthStencilBufferData_Create` at `0x0076E0B0` creates an implicit depth-stencil wrapper from the active D3D depth surface. It is a depth-stencil surface path, not evidence of a sampleable post-process depth texture.
- Render mode global `dword_B42EAC` is set through `sub_7B4890`. The normal world mode is `0`. Observed native offscreen callers temporarily switch to mode `5` for water/cube-map style object work and mode `6` for canopy/shadow work, then restore the previous mode. The debug/console path can also set modes `0..4`.
- Entering render mode `4` calls `sub_7AB1D0(1)`, which builds a table of global `NiScreenElements` quads at `0x00B42CF8..0x00B42D3C` with vertex/z/alpha/stencil/material properties. Leaving mode `4` calls `sub_7AB1D0(0)` and releases them. This is a native debug/special-mode helper, not a Moon Sugar object-mask API.

Render-target conclusion:

- The render target observed at the Moon Sugar hook is the full scene image for that render pass. It is not a small overlay target.
- It is still a flattened color image. The pixel shader sees scene pixels, including object edges already present in the resolved color buffer, but it does not receive NIF object IDs, vertices, materials, bone data, or a sampleable depth buffer.
- A shader can make NIF edges appear to move by deriving an edge mask from color/luma and displacing UVs around those edges. That is the current approach. It remains inferred image-space movement, not actual geometry movement.
- Because the normal interface/cursor pass runs after the world `NiRenderer_Render` call, this hook is better for world-attached drug visuals than a final backbuffer overlay, but it still cannot separate objects from sky, water, particles, or UI-like geometry without another mask/input.

Object draw path:

- `RenderTriShapes` at `0x007694F0` and `RenderTriStrips` at `0x00769700` are the geometry draw points for NIF objects. At these points the engine has the `NiGeometry*`, world bounds, skin state, shader interface, and eventually calls `IDirect3DDevice9::DrawPrimitive` or `DrawIndexedPrimitive`.
- Real vertex/object movement belongs in this family of hooks or in shader/pass generation before these draw calls. A post-process image-space shader cannot move NIF vertices; it can only resample the already-rendered color image.
- Hooking `RenderTriShapes`/`RenderTriStrips` directly for Moon Sugar would be invasive because it would need to preserve every current shader, vertex declaration, stream, index buffer, transform, and render-state assumption around every object draw. It is feasible only as a later controlled experiment, not as a quick shader pass.
- `sub_768890` is the hardware/software skinning gate shared by `RenderTriShapes`, `RenderTriStrips`, `RenderTriGeometries`, and queue-building path `sub_769960`. It checks `BuffData`, `skinData`, hardware bone partitions, mixed vertex processing, and shader-interface capabilities before choosing the skinned path.
- `RenderTriGeometries` at `0x007693E0` is earlier than the final D3D draw. It copies the `NiGeometry` world transform and world bound, chooses the dynamic or unshared geometry group, and calls either `SkinnedRenderTriGeometrySetup` (`sub_767520`) for skinned setup or `NonSkinnedRenderTriGeometrySetup` (`sub_7672F0`) for non-skinned setup.
- `NiGeometry::Render` at `0x00722A10` is not a shader override point. It copies `NiGeometry+0xAC` into `renderer->propertyState`, copies `NiGeometry+0xB0` into `renderer->dynamicEffectState`, then runs geometry controllers. The later renderer path still chooses shaders from the geometry or skin property state.
- The non-skinned renderer setup path `0x007672F0` chooses `NiGeometry::shader` if it casts to `NiD3DShaderInterface`; otherwise it falls back to `renderer->defaultShader`. It then runs the shader-interface vtable sequence around the eventual D3D draw: `+0x28` begin/check, `+0x2C` class-specific render selector, `+0x48` pass-loop begin, then for each pass `+0x30` pass render-state/texture apply, `+0x34` world/skinning setup, `+0x3C` geometry stream/IB binding, `+0x38` `SetupShaderPrograms`, draw, `+0x40` pass cleanup, and `+0x4C` pass-loop advance. `+0x44` is the final shader cleanup after the pass loop.
- The skinned path `0x00767520` is the same ownership model with more moving parts: it selects the shader from the skin/shader property or renderer default, walks hardware/software skin partitions, flushes shader constants, and draws indexed partition buffers. This is why a Moon Sugar mask cannot safely "bind plugin shaders, then call native geometry render" and expect the plugin shaders to remain bound; the native shader interface will overwrite that state before drawing.
- The base `NiD3DShader` vtable clarifies the stream-binding order. Slot `+0x28` (`0x0077A150`) validates render setup and applies render states, `+0x30` (`0x0077A1B0`) applies the current pass render states/texture stages, `+0x34` (`0x00779710`) uploads transforms or skinning state, `+0x3C` (`0x0077A310`) packs geometry when needed and binds `BuffData->VBChip[i]->VB` with `SetStreamSource` plus `BuffData->IB` with `SetIndices`, `+0x40` (`0x00779790`) restores render-state group, and `+0x48/+0x4C` (`0x0077A0B0`/`0x0077A0F0`) own pass iteration.
- The non-skinned transform upload reached from vtable `+0x34` is `sub_765480`. It converts the `NiTransform` passed down from `RenderTriGeometries` into `NiDX9Renderer+0x940` using `sub_761AE0`, whose translation subtracts the current camera-world globals (`CameraWorldTranslate`, `flt_B3F930`, `flt_B3F934`). On pass 0 it also calls `IDirect3DDevice9::SetTransform(D3DTS_WORLD, renderer+0x940)`, updates normal-normalization through render-state vtable `+0x34`, and refreshes renderer model-camera vectors.
- This is now the best-scoped observed non-skinned object-motion hook. A wrapper around `NonSkinnedRenderTriGeometrySetup` (`sub_7672F0`) can copy the stack `NiTransform` argument, apply a small Moon Sugar sway/roll/scale perturbation to the copy, and pass that copy into the original path. That avoids editing NIF transforms, avoids changing culling state, and still lets native shader/pass setup run normally.
- Implementation note: patching the direct call at `0x007694D7` is safer than detouring `sub_765480` itself. IDA shows `sub_765480` has many native helper callers, while the direct call wrapper scopes the effect to normal `RenderTriGeometries` non-skinned object submission. The special path at `0x0076B2E0` also calls `NonSkinnedRenderTriGeometrySetup` (`sub_7672F0`) and is intentionally not patched.
- The wrapper must skip exact `NiScreenElements`, not just test for `NiTriShape`, because `NiScreenElements` derives from `NiTriShape`. The IDA-identified exact RTTI object in the inspected binary is `0x00B3FAA8`; broad `NiTriShape` RTTI is `0x00B3FCD4`.
- `sub_765560` and `NiDX9Renderer::CalculateBoneMatrixes` at `0x007655F0` remain skinned setup evidence. They update renderer `+0x940`, normal-scale state, and for small blend counts write D3D world matrices `D3DTS_WORLD + i` from cached skin bone matrices. They are stale as hook-boundary guidance: the current skinned post-setup hook boundary is `0x007676F9`.
- This gives the manual mask checklist: after collecting visible `NiGeometry*`, a plugin-owned draw must either replicate enough of `0x0077A310` to pack/bind `BuffData` streams and indices or provide a real vtable-compatible shader object that lets the native path do it. The second option is closer to Oblivion's architecture but much larger; the first option is lower-level and must handle missing/stale `BuffData`, skinned geometries, vertex declarations, and state restoration explicitly.
- `NiDX9Renderer::PackGeometryBuffers` at `0x00767B40` and `NiDX9Renderer::PackSkinnedGeometryBuffer` at `0x00763130` are the on-demand `BuffData` owners. They first check whether existing streams are live through `0x00777F10` (nonzero `StreamCount`, every `VBChip[i]` present, every chip has a non-null D3D `VB`). If dirty flags or stream-count mismatch require rebuilding, they release/resize stream arrays through `0x00777F70`, repack vertex data, and rebuild the index buffer through `0x007781F0`.
- The non-skinned pack path uses the active `NiD3DShaderDeclaration` when present. Its `NiDX9ShaderDeclaration` vtable slot `+0x6C` resolves to `0x0076FE20`, which refreshes vertex-buffer chips if the old buffer is missing/too small/force-dirty, sets per-stream stride, writes declaration elements into the mapped buffer, and unlocks the buffer when it owns the lock. Without a declaration, Oblivion falls back to one-stream packing at `0x00776E90`.
- The skinned pack fallback at `0x007774C0` is separate and partition-aware: it writes partition positions, weights, normals, packed colors, and UV data according to the skin partition remap. Its index-buffer rebuild uses the partition's list lengths rather than the base geometry triangle list.
- `0x007781F0` creates or reuses 16-bit D3D index buffers. It only reuses an existing buffer when the descriptor matches the expected type/format/usage/pool and size is large enough; otherwise it calls `CreateIndexBuffer`, locks, copies the source index words, and unlocks. This means a plugin-owned mask pass should prefer already-live `BuffData` for first implementation, and skip stale/missing/skinned/force-dirty geometry until the plugin owns a full packing path.
- `NiGeometryBufferData::RefreshVBChips` at `0x00776C40` is allocation/lifetime ownership, not a getter. It calls the geometry group's `ReleaseChip`, then `CreateChip`, then stores the new `NiVBChip*` back into `VBChip[stream]` if the stream index is valid. A Moon Sugar mask pass should not call this during an effect draw unless it also owns the whole geometry-group packing transaction.
- The safe read-side chip primitive is `0x00761AC0`: it returns `VBChip[stream]` only when `stream < StreamCount`, with no refresh or allocation. This is the useful primitive for conservative live-`BuffData` submission.
- `0x007780A0` and `0x007780D0` confirm that `BuffData` tracks one input layout mode at a time. The FVF setter stores the FVF and releases any cached `IDirect3DVertexDeclaration9`; the declaration setter releases the old declaration, AddRefs the new one, stores it, and clears FVF. A plugin-owned world mask must observe these fields, not mutate them, unless it packed the vertex bytes itself.
- The vertex-buffer manager lock path is also owned. `0x00776C90` locks the D3D vertex buffer, enters a critical section, records the owner thread/source pointer/size, grows a heap staging buffer if needed, and copies locked data into staging. `0x00776D80` copies staging back to the D3D buffer, clears bookkeeping, leaves the critical section, and calls `Unlock`. This reinforces the first-pass rule: do not repack inside the Moon Sugar effect pass.
- The fallback non-declaration layout calculator at `0x00776DD0` starts with a 12-byte position, then appends optional skin weights/indices, normals, color, and UV sets, with UV count clamped by `dword_B28CB0`. It produces FVF-like flags, stride, and offsets for the default one-stream packer. That is useful for recognizing existing layouts, but it is not a license to bind an arbitrary plugin declaration over world buffers.
- `NiDX9RenderState` is cached, not a thin D3D proxy. `SetPixelShader` at `0x0077B080` caches the current pixel shader at `+0xFE8`, `SetVertexShader` at `0x0077B350` caches the current vertex shader at `+0xFE0`, `SetFVF` at `0x0077B410` switches cached input mode to fixed-function and caches `+0x1004`, and `SetVertexDeclaration` at `0x0077B510` switches cached input mode to declaration and caches `+0x100C`.
- Texture and sampler state are cached too. `SetTexture` at `0x0077B280` caches each sampler at `+0xFA0`, `SetSamplerState` at `0x0077B610` caches current/previous sampler-state values, and `ClearTextureList` at `0x0077B210` clears only the cached texture slots. `NiDX9RenderState::Reset` at `0x0077B7B0` rebuilds the render-state cache and writes a set of default D3D render states through the cached setters.
- Consequence for a raw plugin mask draw: a Direct3D state block is acceptable only if it restores the real device state to exactly what Oblivion's `NiDX9RenderState` cache still says is active. If the plugin changes shaders, declarations/FVF, textures, sampler states, or render states and does not restore them, Oblivion may skip necessary rebinds because its cached values did not change. The more native alternative is to route all temporary state through `NiDX9RenderState` cached setters and then restore through those setters, but that requires all needed setter offsets to be correct.
- `CreateDX9ShaderDeclaration` at `0x00771090` allocates a 0x38-byte `NiDX9ShaderDeclaration`, binds it to the renderer/device/vertex-buffer manager, and calls `0x0076E320` to allocate declaration storage. That initializer allocates `StreamCount` stream descriptors, each 0x10 bytes, and for each stream allocates `elementsPerStream` descriptors of 0x1C bytes. Each element is initialized by `0x0076FB00` with sentinel values `0xFFFFFFFF` and `0x11`, meaning disabled.
- The declaration setter path is now mapped. Image-space code calls vtable slot `+0x50` (`0x0076FB30`) as a semantic helper; it maps Oblivion semantic ids to D3D declaration type/usage/index values, then forwards to raw setter `+0x4C` (`0x0076FC40`). The raw setter updates one stream/slot element, marks the declaration dirty at byte `+0x28`, and releases the cached `IDirect3DVertexDeclaration9` at `+0x30`. `0x0076FCF0` disables one element; `0x0076FD60` disables all elements in one stream.
- The internal declaration record is now clearer. `NiDX9ShaderDeclaration +0x1C` is elements-per-stream, `+0x20` is stream count, `+0x24` points to 0x10-byte stream descriptors, `+0x28` is the dirty byte, `+0x2C` is the cached `D3DVERTEXELEMENT9` array, `+0x30` is the cached `IDirect3DVertexDeclaration9`, and `+0x34` is a small boolean flag exposed by slots `+0x64/+0x68`. Each stream descriptor stores an enabled byte at `+0`, its 0x1C-byte element array at `+8`, and computed stride at `+0xC`. Each element stores the computed byte offset at `+4`, source/packing semantic at `+8`, D3D type at `+0xC`, method at `+0x10`, D3D usage at `+0x14`, and usage index at `+0x18`.
- D3D declaration creation happens in `0x00771150`. It walks enabled internal elements, calls `0x00770150` to emit 8-byte `D3DVERTEXELEMENT9` entries and accumulate per-stream byte offsets/stride, appends `D3DDECL_END`, and calls device vtable slot `0x158` (`CreateVertexDeclaration`). This means a plugin-owned world mask shader must match the `BuffData` stream layout used to pack that geometry; binding a different declaration would reinterpret the existing vertex bytes.
- Slot `+0x60` (`0x0076FDD0`) calls the materializer at `+0x74` (`0x00771150`) before returning the computed stride for one stream. Slot `+0x70` (`0x00770120`) materializes and returns the cached D3D declaration plus stream count. These are read/materialize helpers, not safe mutation points during a draw.
- Semantic helper `0x0076FB30` maps TES4 semantic ids to D3D usages: `0` position, `1` blend weight, `2` blend indices, `3` normal, `4` color, `5..0xC` texcoord `0..7`, `0xD` tangent, and `0xE` binormal. Raw declarations can also use special source semantics such as `0x80000000` and `0x80000001`; object shader definitions use those for tangent/binormal-style data that the generic pack dispatcher handles outside the simple semantic switch.
- Concrete object shaders do not share one universal declaration. `Lighting30Shader` at `0x007FC7D0`, `ParallaxShader` at `0x008091D0`, and `SkinShader` at `0x0080AE90` build 6-slot and 8-slot one-stream declarations with position/normal/color/UV plus tangent/binormal-style raw entries. `GeometryDecalShader` at `0x00806210` builds a 2-slot declaration and a 4-slot companion declaration. `ParticleShader` at `0x007E33A0` builds a 5-slot declaration with a special raw element. `WaterShader` at `0x007DD1E0` uses a 2-slot position/UV layout. `PrecipitationShader` at `0x007EEE40` creates a 3-element, 2-stream declaration. This is the decisive reason a single generic Moon Sugar object vertex shader is not state-preserving across normal world geometry.
- Conservative mask rule from declaration ownership: do not force a custom declaration onto world `BuffData` unless the plugin packed the buffers for that declaration. For a first pass, use only already-live geometry whose declaration/stride layout can be positively recognized, or skip it. The native image-space declaration pattern (`CreateDX9ShaderDeclaration(renderer, 2, 1)` with two elements) is valid only for the screen quad path, not arbitrary NIF geometry.
- `BatchRenderGeometries` at `0x0076B310` is only a thunk into `0x00769960`. That target chooses or caches a shader interface, runs the same skinning gate, registers geometry with the dynamic or unshared geometry group, then appends 0x10-byte batch/prepack nodes using the global free list `dword_B42164` and renderer-owned head/tail fields. The non-skinned node is `{NiGeometry*, 0, geomData->BuffData, next}`; the hardware-skinned path emits one node per skin partition with the partition buffer pointer at slot `+8`.
- `NiGeometryGroup::AddGeometryDataToGroup` at `0x00778DB0` is packing registration, not draw submission. For hardware-skinned geometry it calls the group's `AddObject` once per compatible partition; for non-skinned geometry it calls `AddObject` only when `geomData->BuffData` is missing, and returns false when `BuffData` already exists.
- `sub_769030`, called by `NiRenderer_BeginScene` at `0x0076BE00` before D3D `BeginScene`, flushes renderer `PrePackObjects`: it locks and stages shared vertex-buffer ranges, invokes declaration/default/skinned packers, rebuilds index buffers, unlocks, clears the map, and frees temporary entries. This is frame-bound renderer setup, not a reusable mid-frame Moon Sugar object-mask path.
- Conclusion from the batch/prepack decode: do not borrow `PrePackObjects` or `BatchRenderGeometries` for the effect prepass. The lower-risk first implementation remains a plugin-owned visible collection plus manual draw of already-live, non-skinned `BuffData`, with strict state restoration. The batch path is useful as evidence of entry layout and packing ownership, but it is not isolated enough for a second render target pass.
- `RenderTriShapes` and `RenderTriStrips` both call the geometry shader interface update, flush the shader constant manager, branch through skinned partition arrays or `BuffData` arrays, issue `DrawIndexedPrimitive`/`DrawPrimitive`, then clear dirty flags. This is too late to be a simple effect hook; any true wobble here must be state-neutral across all those cases.

Native depth/object paths:

- `sub_7D2070` walks an object list, filters objects by native bitfields, selects shader IDs such as `0x10`, `0x14`, `0x154`, and `0x155`, and calls `NiAVObject_Render`. This is an object depth/ambient-style render path, not the same as the final image-space pass.
- `sub_813300` sets a camera from a node transform/frustum, swaps in a temporary `BSShaderAccumulator`, culls/renders via `sub_7D2070`, then restores the original accumulator.
- `sub_813510` and `sub_813960` set up render-to-target work and call `NiRenderer_BeginScene(kClear_ALL, BSRenderedTexture::UseTextureToRender(...))`. When render mode `dword_B42EAC == 5`, they alter clear color, matching the separate native depth/shadow/water-depth path.
- `BSCubeMapCamera::BSCubeMapCamera` at `0x00814160` builds six face slots, optional render-target fields, a cube frustum, and a private `ImageSpaceShaderList`; it immediately clears that list's nodes. Its dispatcher at `0x00814340` routes mode `0` and mode `3` to the two offscreen target paths, then releases the target and six face references.
- Confirmed callers allocate this camera from texture/offscreen paths such as `sub_7C6310` and `sub_7C6770`. Those paths flip render mode to `5`, update/cull selected objects or lights, and render into engine-owned targets. This is useful as a target-stack and accumulator pattern, but it is not a ready-made "render all world objects into Moon Sugar mask" API.
- The canopy/shadow helper `sub_4D0190` shows the same ownership pattern in another system: allocate temporary `BSRenderedTexture` objects, save renderer clear color/state flags, `StartUsingRenderTarget`, render a shadow scene with `sub_70C0B0`, pop with `0x007D7110`, optionally composite shader ID `0x18`, then restore state and release textures.

Deeper object/prepass ownership:

- `sub_7D2070` is not a generic "render everything to mask" helper. It checks object/shader-property state, consults native bitfields such as `dword_B2C678`, selects internal shader IDs, updates property state, then renders through the active accumulator.
- `sub_813300` is the closest native pattern for a plugin-owned object mask: set a camera from a node transform/frustum, temporarily replace the renderer accumulator, cull with a temporary culling process, call an object-list renderer, submit the accumulator, then restore the original accumulator. Reusing this exact path for Moon Sugar would still require knowing or building the object list and filtering rules.
- `RenderTriShapes` at `0x007694F0` is the final D3D draw path for triangle geometry. It binds the geometry/shader interface, updates shader constants, then calls `DrawPrimitive` or `DrawIndexedPrimitive`. This confirms true NIF vertex wobble belongs in geometry/shader-pass mutation, not in the final image-space pass.
- `sub_7D2070` has only one direct caller found: `sub_813300`. That tightens the conclusion that this is a specialized native offscreen/shadow object pass, not a global draw-list API.
- `sub_7D2070` walks list nodes whose object pointer is stored at node slot `+8`, requires a NiProperty ID `4`, and uses the property's native type to choose hardcoded shader IDs. Types `1..5` select shader ID `0x10` for leaf geometry or `0x14` for nodes with children; type `0x1B` selects `0x154` or `0x155`. The selectors are `sub_7D1320` and `sub_7FD260`.
- If the active accumulator reports pre-existing state through `sub_733840`, `sub_7D2070` bypasses that shader selection and renders the object directly. Otherwise it sets globals such as `dword_B42E90`, updates property state, and then calls `NiAVObject_Render`.
- `sub_7D21F0` is the cleanup/finalize side of this object-list system. It walks the same list, updates eligible shader properties, resets `dword_B42E90`/`dword_B42EB8`, and frees internal lists. This confirms the object list has ownership and cleanup rules specific to the native shadow/offscreen subsystem.

Normal culling and accumulator submission:

- `NiCullingProcess` layout is now clearer: `+0x04` is the `UseAppendVirtual` flag, `+0x08` is an optional visible-geometry array, and `+0x0C` is the active camera pointer. `SceneGraph::SceneGraph` constructs the world culling process with a null visible array, so the normal world path does not automatically provide a reusable "all visible objects" list.
- `NiCullingProcess::Process` at `0x0070E0A0` sets the camera/frustum and temporarily installs an explicit visible array only when one is passed as the fourth argument. If no visible array is present, it prepares the active accumulator with vtable `+0x4C`, traverses `NiAVObject_Render`, then flushes the accumulator with vtable `+0x50`.
- `NiNode::OnVisible` at `0x0070AB40` recurses children when the node bound is nonzero. `NiGeometry::OnVisible` at `0x007227A0` forwards the geometry to `NiCullingProcess::OnVisible`.
- `NiCullingProcess::OnVisible` at `0x0070E1A0` has two distinct modes. With `VisibleGeo != null`, it appends the `NiGeometry*` into a simple visible array. With `VisibleGeo == null`, it calls the active accumulator vtable `+0x58`; if that declines the geometry, it renders the geometry immediately through its render vtable.
- The visible array is only `{ data, count, capacity, grow }`. `sub_732200` resizes the pointer array, and `OnVisible` increments the count. This is useful for a plugin-owned second cull, but it is not a persistent engine-owned object database.
- `sub_70C0B0` has a real non-null visible-array branch, but all observed native xrefs pass `visibleGeo = 0`. The normal world calls, canopy/shadow paths, MiscPass, and shadow-light/cubemap-style paths rely on either the culling process's existing `VisibleGeo` pointer or accumulator-direct traversal. The explicit visible-array route appears available to a plugin, but it is not exercised by the decoded stock callers.
- Passing a visible array to `sub_70C0B0` is not collection-only. That wrapper zeros the count, calls `NiCullingProcess::Process(..., visibleArray)`, then immediately calls `sub_70C000(camera, visibleArray)`, which prepares the active accumulator, consumes the visible list through `sub_70BF30`, and flushes. A plugin that only wants to collect a list should call `NiCullingProcess::Process` directly with its explicit array; a plugin that uses `sub_70C0B0` should already have its mask target and render/accumulator state in place.
- `sub_70BF30` is only a visible-array submitter. If `g_Renderer->accumulator` is present, it AddRefs that accumulator and calls vtable `+0x54` with the visible array; if no accumulator is present, it renders every listed `NiGeometry` directly through the geometry render vtable `+0x84`.
- `BSShaderAccumulator` vtable `+0x54` (`0x0071AC10`) does not render a plugin-neutral mask. It iterates the visible array, direct-renders geometry whose render flags disqualify it from accumulation, and otherwise appends the geometry into the accumulator's normal grouped lists for later material-owned flush. This confirms that a second cull can reuse Oblivion visibility, but the stock visible-list consumer still routes back into native geometry/material drawing.
- `BSCullingProcess` is not automatically safer than the base class. Its vtable `ProcessCull` override at `0x006FBB80` checks byte/field `+0x90`: if nonzero it calls the normal frustum test path, but if zero it bypasses frustum testing and calls `OnVisible` directly. `MiscPass` explicitly sets this field to zero. For a Moon Sugar second cull, base `NiCullingProcess` or `BSCullingProcess` with culling enabled is the state-preserving candidate; using the bypass mode would over-collect objects outside the camera frustum.
- `BSCullingProcess::~BSCullingProcess` at `0x0070DFA0` only restores the base `NiCullingProcess` vtable. It does not free a visible array or other heap state, so a plugin-owned visible array must own and free its own `data` buffer.
- The relevant `BSShaderAccumulator` vtable entries are now mapped:
  - `+0x4C` -> `0x007A9800`, prepare with camera; stores the camera at accumulator `+0x08` and sets pending flag `+0x2268`.
  - `+0x50` -> `0x007AE070`, flush/sort/dispatch queued render passes and special lists.
  - `+0x54` -> `0x0071AC10`, consume an explicit visible array and either direct-render or enqueue each geometry.
  - `+0x58` -> `0x007AD270`, per-geometry enqueue path used by normal culling when no visible array exists.
- `BSShaderAccumulator::OnVisible` at `0x007AD270` starts from the `NiGeometry` pointer and reads `NiGeometry+0xAC` through `sub_405760`; that field is a refcounted render-pass bundle, not a plugin-neutral material handle. If no bundle exists, the geometry is placed into an immediate geometry group through `sub_7ABC40` and appended with `sub_7ABDE0`. If a bundle exists, the function classifies the bundle's render passes by native shader/pass id, appends them into accumulator buckets at `accumulator + 0x14 * (passId + 0x0D)`, or emits through special paths.
- `sub_7ABC40` allocates a paired group containing a `NiGeometry*` list and a `BSShaderProperty::RenderPass*` list, then links that group under the accumulator's immediate-geometry list. `sub_7ABDE0` is only a tail-append helper for the persistent lists. These helpers are useful for understanding queue ownership, but they do not provide a way to replace the draw shader.
- `sub_7AC9A0` is the flush-side owner for a single shader/pass bucket. It normalizes a few hardcoded ids, refreshes native shader constants through `sub_7D1320` or `sub_7FD260`, sorts the bucket, and calls `sub_7A9820(renderPass, passId)` for each queued render pass unless the pass id belongs to a native special/deferred range such as `0x156..0x158` or `0x160..0x162`.
- The accumulator queue is therefore shader-property owned. A plugin can observe this path to know which world geometry is visible, but drawing a Moon Sugar mask by forcing pass ids through the normal accumulator would couple the effect to native `BSShaderProperty::RenderPass` state, global shader id state (`dword_B42E90`/`dword_B42EB8`), and pass-specific constant setup. The lower-risk path is still a plugin-owned render target plus a plugin-owned flat mask/distortion shader, fed either by a second explicit visible-array cull or by a tightly gated per-geometry hook that renders to the plugin target outside the native accumulator buckets.
- This corrects the earlier mask/prepass assumption: the normal world render path is usually accumulator-direct, not cull-then-reuse-list. A Moon Sugar object mask has two realistic native shapes: run a second cull into a plugin-owned visible array and then render that list into a mask target, or hook the accumulator-direct `+0x58` path while a plugin mask target is active. The second cull is slower but lower risk; the accumulator hook appears closer to one-pass visibility but is far more invasive.
- `EndTargetGroup` at `0x007AAA30` reinforces the same ownership model. It processes accumulator-owned special/refraction lists into `dword_B474AC` using hardcoded shader IDs and object property flags, then restores accumulator and clear-color state. This is a useful architectural precedent, not a generic mask API Moon Sugar can call blindly.
- `MiscPass` at `0x0057F170` constructs a temporary `BSCullingProcess` with `sourceCullingProcess->VisibleGeo`, then later processes the scene with that temporary object. This confirms visible arrays are opt-in/inherited culling state; when the source culling process has no visible array, this path inherits null and uses accumulator-direct traversal too.
- The canopy/shadow path at `0x004D0190` is the closest low-level offscreen precedent for a Moon Sugar mask render target. It allocates temporary rendered textures, constructs a culling process with null `VisibleGeo`, enters targets with `StartUsingRenderTarget`, calls `sub_70C0B0`, pops with `sub_7D7110`, and restores renderer state. It still relies on native shader IDs and scene-specific flags, so it is a pattern, not a reusable mask API.
- Native offscreen object passes force alternate shaders through `sub_7A9820`, `sub_7D1320`, and `sub_7FD260`. `sub_7A9820` sets `dword_B42E90` to the requested shader/pass ID and `dword_B42EB8` to the active render-pass object before calling the object render vtable. `sub_7D1320` and `sub_7FD260` update native shader program/pass render-state selections for hardcoded IDs. This is how Oblivion gets flat/depth/special object renders, but hijacking those IDs for Moon Sugar would collide with renderer-owned shader definitions.
- Build 78 follow-up: the same object/offscreen family still does not expose true identity masking. `sub_7D2070` has object pointers while walking its caller-owned list, but it routes to native shader IDs and `NiAVObject_Render`. `TESObjectCELL::CreateCanopyShadowMaskForCell` (`0x004CE460`) and `ShadowCanopyPass` (`0x004826F0`) create and consume a cell canopy-shadow texture, not an object/form/material ID buffer. The pass-name table at `0x007B4920` names depth, render-normal, local-map, land, grass, water, and refraction passes, but no inspected pass is validated as a terrain material ID, object ID, actor ID, form ID, or plugin-neutral identity channel. The decode artifact is `ida_decodes/build78_true_identity_masking/true_identity_masking_decode.md`.
- Build 79 follow-up: the plugin now records read-only mask/prepass candidate counters from the already-hooked world render wrapper. The counters only report whether `sub_70C0B0` calls use the current world camera, current world scene graph, scene graph culling process, normal render mode, a non-null fourth `visibleGeo` argument, or a non-null `NiCullingProcess::VisibleGeo` pointer at offset `+0x08`. This is runtime evidence collection for a future plugin-owned prepass; it does not collect a visible list, render a mask target, or add terrain/object identity semantics. The decode artifact is `ida_decodes/build79_plugin_mask_prepass/plugin_owned_mask_prepass_contract.md`.

Native refraction path:

- `GetShaderDefinition(0x15)` constructs the native `RefractionShader` only when `ShaderPackage >= 2`; `ImageSpaceshaderList::Create` adds it only when `enableRefraction` is enabled.
- Refraction active predicate `0x008002E0` is simply `dword_B474AC != 0`. That global is a transient `BSRenderedTexture*`, not persistent gameplay state.
- `sub_800B30(char create)` lazily allocates `dword_B474AC` from default render-target type `0x14`. Type `0x14` is full default-render-target width/height, usage `0x46`, format `0`.
- `EndTargetGroup` paths at `0x007AAB08` and `0x007AADDC` allocate/use `dword_B474AC` while the renderer has queued refractive geometry. This is the native bridge from object-marked refraction to the later image-space composite.
- `RefractionShader` loads `imagespace\1x\v\base.v.hlsl` with `TEX2` and `imagespace\2x\p\refraction_P20.p.hlsl` in `sub_800750`. Its render pass at `0x00800440` binds the current source texture and the `dword_B474AC` refraction texture, draws a screen-space pass, then releases `dword_B474AC`.
- `sub_802890` stores the primary source `BSRenderedTexture` on `BSImageSpaceShader` offset `+0x7C` with refcounting. Refraction then stores `BSRenderedTexture::GetInnerTexture(dword_B474AC)` on its own extra texture slot at offset `+0xA0`.
- Refraction constant setup `0x008002F0` exposes vertex constants `texRatio0` and `texRatio1`, plus pixel constant `Ambient Color`. Pass setup `0x00800900` creates two texture stages before applying the same broad image-space state disables. This is direct Oblivion evidence that a two-input image-space pass is native-compatible when the shader class owns and binds the second texture.
- Refraction ownership starts on objects/materials. `SetTargetRefraction` at `0x00507E60`, `SetTargetRefractionFire` at `0x00507FF0`, and `SetActorRefraction` at `0x0050E340` route through actor/process transparency or `sub_7D92C0`.
- `sub_7D92C0` walks a `NiNode` tree and toggles shader-property bits `0x8000` and `0x10000`, storing refraction power and optional period fields. The NIF/material parser also recognizes `Refract`/`RefractF` at `0x007DA5C0` and sets these flags.
- Conclusion: refraction is the strongest Oblivion-native example of object-assisted image-space distortion, but it is object/property owned. Moon Sugar should not hijack object refraction flags globally. The useful pattern is architectural: render a second plugin-owned mask/distortion texture, then composite it with the scene in one image-space pass.

Practical conclusion:

- The current hook is correctly placed for a visible post-process result but only receives resolved color. It cannot know exact NIF object IDs, per-object depth, or vertex positions.
- A lower-risk candidate upgrade is a two-input image-space pass: keep the final color texture as sampler `s0`, then create a lightweight Moon Sugar mask/depth-ish prepass as sampler `s1`. That prepass would need to render eligible world geometry as high-contrast silhouettes or depth ramps before the Moon Sugar shader could limit motion to masked object edges instead of all high-contrast UI/world edges.
- The lower-risk first step is not full real geometry deformation. It is to allocate one plugin-owned render target, run a second cull into a plugin-owned visible array or equivalent isolated mask submission, then render a silhouette/depth-ish mask with plugin-owned D3D state. Reusing `NiGeometry::Render` for that mask would require a temporary vtable-compatible `NiD3DShaderInterface` or risky geometry/default-shader pointer swaps, because native geometry render owns shader selection and pass setup. A manual mask submission must instead own stream/index binding and restore D3D state tightly before binding that mask to sampler `1` in `DrawDistortionQuad`. The conservative first version should draw only non-skinned `NiGeometry` whose `geomData->BuffData` already has live streams and an index buffer, and skip geometry that would require pack/repack.
- A true object-space drug effect would require a later geometry-render hook around `0x007694F0`/`0x00769700` or shader-property/pass mutation, with strict save/restore and object filtering. That path can make actual NIFs wobble, but it is substantially higher crash/regression risk.

## Renderer Decode: Image-Space Enhancement Candidates

This pass focused on engine-native levers that can make Moon Sugar feel more like a world/camera drug effect instead of only a color-buffer overlay.

Native image-space list:

- Renderer/shader initialization at `0x007BA0F0` constructs `imageSpaceShaderList` at global `0x00B42D7C` when `ImageSpaceEffectEnabled` is enabled.
- `ImageSpaceshaderList::Create` at `0x00803A30` builds the native list in this order:
  - shader ID `0x12`: NightEye
  - shader ID `0x8` if HDR is enabled, otherwise `0x7`: HDR or Blur
  - shader ID `0x15`: Refraction, only when `enableRefraction` is enabled
  - shader ID `0x0B`: Hit/double-vision
  - shader ID `0x0C`: Copy fallback stored separately at list offset `+0x18`
- `ProcessImageSpaceShader` uses vtable `+0xAC` as each shader's active predicate and vtable `+0xA8` as its render pass. Constructors call vtable `+0xB0` to load shader programs and `+0xB4` to build D3D passes.
- `AddImageSpaceShader` at `0x00803790` appends a `BSImageSpaceShader*` node to the list. It does not visibly increment the shader refcount, so a plugin-owned synthetic shader object would need stable lifetime and careful handling across `RecreateImageSpaceShader` at `0x007BA418`.

Observed native image-space gates:

- NightEye active predicate `0x007F4F70` returns true when `flt_B46914 > 0.0`. Constant setup at `0x007F4FA0` maps the pixel constant named `spellinput` directly to `flt_B46914`.
- Hit/double-vision active predicate `0x007EB0E0` returns true when `flt_B46124 > 0.0` or `flt_B46120 > 0.0`.
- Blur active predicate `0x007E9C00` returns `byte_B43073`.
- Refraction active predicate at `0x008002E0` returns true when `dword_B474AC != 0`.
- Native image-space shader files include examples under `imagespace\1x\p\` and `imagespace\2x\p\`, including `nighteye.p.hlsl`, `copy.p.hlsl`, `blur_20.p.hlsl`, and `doublevis_P20.p.hlsl`. This is direct evidence that Oblivion already has TES4-era blur/double-vision image-space machinery.

Ownership pass:

- `byte_B43073` is Bloom Lighting renderer state, not an effect timer. It is set/cleared during `CreateWindowAndInitialize` at `0x004987E4-0x004987ED` and reported as `Bloom lighting` at `0x00498C23`.
- Native blur is configured through `sub_7B4830(int blendType, int passCount, float radius, float alpha, float sunOrSky, int texSize)`. WinMain initializes it from `BlurShader` INI settings (`iBlendType`, `iNumBlurpasses`, `fBlurRadius`, `fAlphaAddInterior`, `fAlphaAddExterior`, `iBlurTexSize`), `SetImageSpaceGlow` calls it in the non-HDR path at `0x00508963`, and SunDamage update/remove paths also call it. Do not use these globals as Moon Sugar-owned state.
- `BlurShader_P20` loads `copy.p.hlsl`, `blur_20.p.hlsl`, `blend_P20.p.hlsl`, and a debug copy pass in `sub_7EA510`. Its render pass at `0x007E9DB0` allocates default render target type `0x10`, then ping-pongs the scene through the native blur passes using the configured radius/pass globals. This is a good reference for TES4-era blur cadence.
- Native hit/double-vision is configured by `sub_7B4870(float hitOffset, float blockedOffset, float blurAmount)`, initialized from `GethitShader` INI settings `fHitTexOffset`, `fBlockedTexOffset`, and misspelled `fBlurAmmount`.
- `sub_7EB010(char blocked)` is the native hit trigger. It is gated by `byte_B2D91C`; when active it raises `flt_B46124` to either the normal hit or blocked-hit offset and writes `flt_B46120` to the secondary blur amount. The only direct gameplay caller found so far is player `Actor_AttackHandling::Blocking` at `0x00600572`.
- Native `HitPass` at `0x007EB3D0` allocates default render target type `0x13`, decays `flt_B46124` and `flt_B46120` every frame, writes sin/cos double-vision offsets to `0x00B2D90C/0x00B2D910`, writes texture ratios to `0x00B2D914/0x00B2D918`, and feeds those through the `fDoubleVis` constant map created at `0x007EB250`. It loads `copy.p.hlsl`, `blur_20.p.hlsl`, and `doublevis_P20.p.hlsl` in `sub_7EBC80`.
- `byte_B2D91C` is enabled by a menu/dialog/player-state restore path at `0x0059DF5B` and cleared by reset/cleanup at `0x0059ED3F`, which also zeros `flt_B46124` and `flt_B46120`. The RaceSex/menu reset path at `0x0057B894` zeros those hit globals as well.
- NightEye globals are owned by the NightEye actor-value path. `NightEyeEffect_SetPlayerShader?` at `0x006A2EA0` reads player actor value `kActorVal_NightEyeBonus` and calls `sub_7F4DE0` to set or clear `flt_B46914`. Do not hijack NightEye state for Moon Sugar.

Do not blindly write these native globals. They are shared renderer, magic, combat, command, and UI state. The safer Moon Sugar path is to mirror the useful native behavior in plugin-owned constants: time-decayed double vision, small sin/cos offsets, restrained blur, and TES4-style color pressure.

Image-space list lifetime:

- `ImageSpaceShaderList::ProcessImageSpaceShader` at `0x008037D0` creates a temporary list of active shaders, binds each active shader to the shared `NiScreenElements`, refcounts that binding, and ping-pongs source/destination render textures. The render list itself is not a permanent ownership mechanism for plugin objects.
- `ImageSpaceShaderList::Destroy` at `0x00803690` calls `NiTPointerList::FreeAllNodes`, releases the cached full-size ping-pong target, clears fallback pointer `+0x18`, releases the shared screen quad, then destroys the pointer list. It does not release shader objects stored as list-node data. Stock image-space shader object ownership lives in `ShaderDefinition`, not in the list nodes.
- `AddImageSpaceShader` at `0x00803790` only allocates a node, writes the `BSImageSpaceShader*`, links it, and increments `numItems`; no shader refcount is visible. A plugin-owned object inserted here must outlive the list and must not rely on the list to release it.
- `sub_803570` at `0x00803570` is the one-off image-space helper used by menu/offscreen paths. It ensures the quad, binds one shader temporarily, calls that shader's render method, then releases the quad's shader binding. It is a useful calling pattern, not a persistent ownership path.
- `RecreateImageSpaceShader` at `0x007BA418` is inside the function starting at `0x007BA2F0`. When the shader system is already initialized, it calls shutdown helpers, clears shader maps, destroys and frees `imageSpaceShaderList`, deletes and frees `g_textureManager`, rebuilds the texture manager, and recreates `imageSpaceShaderList` if `ImageSpaceEffectEnabled` is still enabled.
- A plugin-owned synthetic `BSImageSpaceShader` inserted into the native list would need stable allocation, teardown, and recreate handling. It would also need to survive the fact that the global texture manager can be deleted and recreated under `RecreateImageSpaceShader`. The current call-site post-pass remains lower risk.

Recreate and texture-manager lifetime:

- `RecreateImageSpaceShader` at `0x007BA2F0` is called from the display/render resize path `0x00497E70` after the renderer dimensions are recreated. The normal first-time path is still `sub_7BA0F0`, called from scenegraph/renderer startup at `0x00406A83`.
- The recreate path does not visibly free or zero the `ShaderDefinition` cache at `0x00B42EC0`. Instead, it first calls `sub_7B84E0`, which walks cached definitions from `B42EC0` to `B42F30`, asks each shader for its type through vtable `+0x1C`, skips type `-1`, then calls shader vtable `+0x90`.
- The exact display-resize resource fence is `sub_497E70` at call site `0x00497FE9`: it calls `RecreateImageSpaceShader(a1)` after `sub_76B090` has recreated the Gamebryo render dimensions and before a new shader accumulator is installed and the world/menu/interface scenegraphs are rebuilt. This is the clearest native place for a plugin to invalidate/rebuild renderer-owned helper resources after a resize.
- For image-space shaders, vtable `+0x90` is a pass texture cleanup selector: Copy/Blur use `sub_801420` to clean the inherited pass slot, NightEye/Refraction use `sub_800B10` to clean their concrete pass slot, Hit uses `sub_7EC7A0` to clean two pass slots, and HDR uses `sub_7C02B0` to clean its larger pass array.
- The actual stage cleanup target is shared vtable `+0x94`, `sub_801430`. It iterates a pass's texture stages, calls `sub_75FB10` to get and temporarily `AddRef` the stage texture, preserves the global texture `dword_B43110`, and otherwise releases and clears the stage texture pointer. This is a targeted texture-reference detach, not a shader object destructor.
- `sub_7B84E0` is pass texture-stage cleanup, not shader-program handle restore. It does not walk arbitrary plugin-owned `NiD3DVertexShader` wrappers and does not call vertex wrapper vtable `+0x5C`.
- `sub_7B84E0` is not only a display-reset helper. It is also called during TES cleanup/streaming at `0x0043FC6A`, cell transition/player-node handling at `0x004456E5`, and `TESObjectREF_SetCastShadows` at `0x004DB391`. This marks it as the engine's general "cached shader pass texture refs may now be stale" cleanup pass.
- After that detach, recreate clears two shader maps through `dword_B42EBC + 0xC` and `+0x8`, releases two global texture refs through `sub_7E30F0` and `sub_7F3BA0`, destroys `imageSpaceShaderList`, deletes `g_textureManager` through `BSTextureManager_Delete`, recreates the manager, prewarms/returns common targets through `sub_7C2450`, recreates global screen-element quads through `sub_7AB1D0(1)`, and finally rebuilds `imageSpaceShaderList`.
- `BSTextureManager_Delete` at `0x007C2100` releases available/in-use rendered textures, shadow maps, cached buffers, and internal list payloads. `sub_7C2450` allocates or retrieves common HDR, image-space, and default render targets, then returns them to the manager cache with `sub_7C1EE0`; it is a render-target cache warmup, not a one-off Moon Sugar target allocation API.
- Consequence for a candidate native Moon Sugar shader: any native object inserted through the stock list should expect cached shader objects to outlive list rebuilds while their pass texture-stage refs are scrubbed. A plugin-owned pass, replacement vertex wrapper, or secondary mask texture must either participate in the same cleanup/recreate discipline or be fully outside the native manager lifetime with its own reset handling. For the whitelisted object-wobble replacement, treat `0x00497FE9`/`0x007BA2F0` as the practical rebuild boundary for plugin-owned wrappers, not as a guarantee that Oblivion will restore them.

Concrete image-space class pass:

- Stock image-space vtables confirm the common slots used by the list: `+0xA8` is the render pass, `+0xAC` is the active predicate, `+0xB0` loads shader programs, and `+0xB4` builds D3D passes. The `+0x8C` wrapper often reloads/rebuilds; `+0x90/+0x94` select and clean pass texture stages during stale-resource cleanup.
- `CopyShader` is not a normal active-list shader. Its active predicate is `nullsub_returnTrue_0arg`, but `ImageSpaceshaderList::Create` stores it separately at list offset `+0x18` as the fallback used only when no active shaders exist. A Moon Sugar shader should not copy this "always active" predicate unless it is intentionally replacing the fallback path.
- `CopyShader::Render` at `0x00803E40` is the cleanest single-sampler render reference. It computes source and destination ratios, writes fields at `+0xA8..+0xB4`, writes the mode field at `+0xA4`, stores the source through `sub_802890`, sets a screen-space viewport, and draws the shared quad.
- `sub_802890` at `0x00802890` is the base source setter: it refcount-stores the current `BSRenderedTexture*` in `BSImageSpaceShader +0x7C`. `CopyShader`/NightEye/Refraction all use this before drawing.
- `sub_8048E0` is the concrete primary texture binding point for Copy-style image-space passes. It clones the first pass texture stage, calls `BSRenderedTexture::GetInnerTexture(*(this + 0x7C))`, and writes that `NiRenderedTexture*` into the stage with `sub_76C910`. This confirms the source scene is sampled through normal texture-stage state, not through an external overlay convention.
- `NightEyePass` at `0x007F5120` is the simplest active shader body. Its active predicate at `0x007F4F70` is `flt_B46914 > 0.0`; it computes texture ratios, stores the source at `+0x7C`, begins the destination/default target, sets the screen-space camera, and draws one quad. Its loader uses `imagespace\1x\v\base.v.hlsl` as `vs_1_1` and `imagespace\1x\p\nighteye.p.hlsl` as `ps_1_3`.
- `RefractionShader` is the clearest two-sampler model. Its active predicate at `0x008002E0` is `dword_B474AC != 0`; its loader at `0x00800750` uses `base.v.hlsl` with the `TEX2` macro and `imagespace\2x\p\refraction_P20.p.hlsl`; its pass builder creates two texture stages. The render pass at `0x00800440` stores the scene source at `+0x7C`, binds `dword_B474AC` as the secondary rendered texture at `+0xA0`, draws, then returns/releases `dword_B474AC` through the texture manager and clears the global.
- `GethitShader` is a larger three-stage double-vision/blur chain. Its active predicate at `0x007EB0E0` is `flt_B46124 > 0.0 || flt_B46120 > 0.0`; its loader at `0x007EBC80` uses `copy.p.hlsl`, `blur_20.p.hlsl`, and `doublevis_P20.p.hlsl`; its pass builder creates multiple passes with one or two stages. `HitPass` at `0x007EB3D0` uses temporary render-target type `0x13`, decays the hit globals, writes sin/cos offset constants, and runs the double-vision chain.
- Consequence for a candidate native Moon Sugar shader: the closest observed native shape is not Copy fallback. It is a NightEye-style active predicate and source pass, optionally extended with the Refraction two-stage pattern if a plugin-owned mask or history texture is introduced. The Hit shader remains the best TES4-era behavioral reference for restrained double vision, but its globals and gameplay ownership should remain untouched.

Stock render-pass cadence:

- `BSImageSpaceShader::RenderShader` at `0x008028E0` uses the render-target stack path: `StartUsingRenderTarget`, source store at `+0x7C`, `SetupScreenSpaceCamera(0)`, shared quad draw, then `sub_7D7110` to pop/resume the previous target. This is the base fallback style.
- Concrete Copy, NightEye, and Refraction render methods instead use the direct begin/draw/end style. They compute source/default or source/destination ratios, call `sub_802890` to own the source at `+0x7C`, begin either the supplied destination via `BSRenderedTexture::UseTextureToRender` or the default target via `NiRenderer_BeginScene1`, set a viewport-backed screen-space camera, submit the quad, and call `NiRenderer_EndScene` only when they opened an explicit destination pass.
- `CopyShader::Render` at `0x00803E40` is slightly special because it can use global override sources `dword_B474F0` or `dword_B474EC`; those set mode field `+0xA4` to `1` or `2` and skip the normal `+0x7C` source store. Normal image-space copy sets mode `0` and stores the scene source.
- `NightEyePass` writes ratio/texel vectors into the fields exposed by its constant map (`texRatio0` and `texRatio1`) before drawing. It is the cleanest one-pass active effect reference because it has one source texture, one active predicate, and no secondary temporary targets.
- `RefractionPass` mirrors NightEye for the primary scene source, then binds `BSRenderedTexture::GetInnerTexture(dword_B474AC)` into its extra texture slot at `+0xA0`. After the draw it releases that extra texture reference, returns `dword_B474AC` through `sub_7C1EE0`, releases the global, and zeros it. This confirms the native two-input object-assisted composite has strict one-frame ownership.
- `HitPass` is the native reference for drug-like double vision: it allocates two temporary type `0x13` render targets, runs copy/blur intermediates, decays the hit strength globals by frame time, writes sine/cosine double-vision offsets into `fDoubleVis`, composites with `doublevis_P20`, clears pass texture references, and returns both temporary targets to the texture manager.
- Target helpers clarify the lifetime boundary. `NiRenderer_BeginScene`/`NiRenderer_BeginScene1` set separate scene-state flags and call `StartUsingRenderTarget`; `NiRenderer_EndScene` drains all render-target stack entries through `sub_7D7150` before ending the D3D scene when appropriate. A Moon Sugar native pass must either stay inside this discipline or keep using its current isolated D3D state-block approach.

Native image-space quad submission:

- `sub_709C60` at `0x00709C60` is only a three-instruction virtual thunk: it loads the `NiScreenElements` vtable entry at `+0x84` and jumps to it. The useful detail is in the callers: native image-space render methods push `g_Renderer` before calling this thunk (`BSImageSpaceShader::RenderShader` at `0x00802984`, `CopyShader` at `0x00804184`). The renderer argument is therefore preserved on the stack through the thunk.
- The `NiScreenElements` vtable `+0x84` target resolves to `sub_717690`. It calls `NiGeometry::Render(this, renderer)`, which copies the geometry property and dynamic-effect state into the renderer, then the active-scene path calls the renderer vtable at `+0x160`.
- The `NiDX9Renderer` vtable at `+0x160` resolves to `RenderTriGeometries` at `0x007693E0`, not the final `RenderTriShapes` shortcut. This corrects the earlier assumption that the image-space quad jumps directly to a raw draw path. Native image-space quads still go through the normal geometry shader setup path.
- For the screen quad, `RenderTriGeometries` reaches the non-skinned shader setup path `NonSkinnedRenderTriGeometrySetup` (`sub_7672F0`). That path calls the shader vtable begin/check method `+0x28` (`0x0077A150`), then the class-specific selector `+0x2C`, then enters the pass loop through vtable `+0x48` (`sub_77A0B0`, select first pass). Each loop applies pass render-state/texture stages through `+0x30` (`0x0077A1B0`), uploads non-skinned world state or skin state through `+0x34` (`0x00779710`), packs/binds geometry streams through `+0x3C` (`0x0077A310`), applies program/constant setup through vtable `+0x38` (`SetupShaderPrograms` at `0x0077A1F0`), draws, restores shader render-state group through `+0x40` (`0x00779790`), then advances/restores passes through `+0x4C` (`0x0077A0F0`).
- `SetupShaderPrograms` calls the current pass setup `sub_75FBA0`. That function applies the pass pixel and vertex shaders through `renderStateG`, applies pixel and vertex constant maps, and sets the vertex declaration or fixed-function FVF. `NiD3DPass::ApplyPassToRender` at `0x0075FD90` is reached earlier through shader vtable `+0x30`; it sets software vertex processing, applies the pass render-state group, applies each texture stage through `sub_772400`, and disables unused texture stages through `sub_771790`.
- The texture-stage apply path reaches `sub_7721E0`, which resolves the `NiTexture`/`NiRenderedTexture` through the DX9 texture manager, then calls the render state `SetTexture(stage, resolvedD3DTexture)` and applies sampler/texture-stage state. The native image-space source is therefore a real D3D sampler binding, not a side overlay convention.
- `CopyShader` shows the primary source texture binding point. Its vtable `+0x2C` target `sub_8048E0` clones the pass texture stage, calls `BSRenderedTexture::GetInnerTexture(*(this + 0x7C))`, and writes that `NiRenderedTexture*` into the texture stage through `sub_76C910`. `sub_76C910` is a refcounted `NiD3DTextureStage` texture setter. This is how the `BSImageSpaceShader +0x7C` scene source becomes sampler input for the native full-screen pass.
- Consequence for Moon Sugar: the current plugin-owned D3D quad is still architecturally close to native image-space rendering because it binds the full-scene source texture and draws in screen space. A truly native synthetic shader, however, must reproduce the object/vtable/pass setup path above: source ownership at `+0x7C`, begin/setup method that binds the inner texture to a `NiD3DTextureStage`, pass construction, program selection, constant maps, and pass cleanup.

Native screen-quad data and declaration:

- `NiScreenElementsData::Construct` at `0x00703230` allocates polygon, vertex, optional normal/color, optional texture-coordinate, and triangle-index storage. The image-space list calls it from `0x00803442` with one polygon, four vertices, one texture set, and index capacity for two triangles.
- `sub_702970(screenData, 4, 0, 0)` appends one four-vertex polygon. With `a3 == 0`, it creates the default triangle fan indices itself: `(base, base + 1, base + 2)` and `(base, base + 2, base + 3)`.
- `sub_702EC0(screenData, 0, -1.0, 1.0, 2.0, -2.0)` writes the fullscreen positions. The four vertices become `(-1, 1, 0)`, `(-1, -1, 0)`, `(1, -1, 0)`, and `(1, 1, 0)`.
- `sub_702FC0(screenData, 0, 0, 0.0, 0.0, 1.0, 1.0)` writes the texture rectangle for texture set zero. The four UVs become `(0,0)`, `(0,1)`, `(1,1)`, and `(1,0)`.
- `CreateDX9ShaderDeclaration` at `0x00771090` allocates a renderer-backed `NiDX9ShaderDeclaration`, stores the renderer/device/buffer-manager references, and prepares one stream with two declaration slots for image-space shaders.
- Copy, Hit, NightEye, and Refraction all build the same two-element declaration: `AddEntry(0, 0, 0, 2, 0)` and `AddEntry(1, 7, 5, 1, 0)` through vtable `+0x50` (`sub_76FB30`). Decoding through `sub_76FB30`, `sub_76FC40`, and `sub_770150` maps this to a position-like `FLOAT3` element and a `FLOAT2 TEXCOORD0` element. The concrete D3D declaration is created lazily by `sub_771150`, which writes `D3DVERTEXELEMENT9` entries and calls the D3D device `CreateVertexDeclaration`.
- This means a native Moon Sugar shader should not invent a different fullscreen vertex format. The era-accurate path is to use the same position plus texcoord declaration and the same `NiScreenElements` geometry, then put the drug behavior in shader constants and sampler binding.

Image-space shader object layout:

- `NiD3DShader::NiD3DShader` at `0x0076D040` is the lower base constructor used before `BSShader`. It clears `IsInitialized`, declaration, render-state group, shader-level pixel/vertex constant maps, current pass, pass count, and creates a `NiTArray<NiPointer<NiD3DPass>>` with initial/grow size `2`.
- `NiD3DShader::~NiD3DShader` at `0x0077AA80` unregisters the shader from the shader factory/renderer when needed, clears render-state ownership, releases every pass in the pass array, releases `CurrentPass`, releases shader-level vertex and pixel constant maps, releases the shader declaration, releases the D3D device reference, and tears down the underlying shader-interface base.
- `NiD3DShader_RemoveShaderPasses` at `0x0077AA10` is the scoped pass cleanup helper. It releases each pass entry, releases `CurrentPass`, and zeros `PassCount`/`CurrentPassIndex`.
- `BSShader::BSShader` at `0x00802540` extends `NiD3DShader`, installs the `BSShader` vtable, constructs one local pass pointer slot at `+0x70`, clears it, and clears a byte at `+0x78`.
- `BSShader::~BSShader` at `0x008012E0` releases the `+0x70` pass slot, unregisters from the shader interface list, destructs the slot, then calls the `NiD3DShader` teardown path.
- `BSImageSpaceShader::BSImageSpaceShader` at `0x008026C0` calls the `BSShader` constructor, installs the `BSImageSpaceShader` vtable, clears the source `BSRenderedTexture*` at offset `+0x7C`, and clears scalar fields at `+0x80..+0x8C`.
- `BSImageSpaceShader::~BSImageSpaceShader` at `0x00802760` releases the source texture at `+0x7C`, clears `+0x80..+0x8C`, then calls `BSShader::~BSShader`. The base class therefore owns only the primary scene source pointer and common shader/pass fields.
- Native image-space vtable slots are now directly confirmed:
  - `+0x84`: generic renderer/pass initialization (`sub_8025F0`)
  - `+0x8C`: load/build wrapper for some classes, or class-specific cleanup/setup
  - `+0x90`: class-specific pass cleanup selector used by `sub_7B84E0`
  - `+0x94`: shared pass texture-stage cleanup helper (`sub_801430`)
  - `+0xA8`: render pass
  - `+0xAC`: active predicate
  - `+0xB0`: shader program load
  - `+0xB4`: D3D pass construction
- `sub_8025F0`, the generic vtable `+0x84` initializer, binds `dword_B43104` as renderer through `NiD3DShader::SetRenderer`, calls virtual setup/check slots `+0x98`, `+0xA0`, and `+0x9C`, then grows pass storage for one pass through `sub_76CCA0`. Stock image-space definition builders call this before program load/build.
- `NiD3DShader::SetDeclaration` at `0x0077A400` refcount-stores the declaration on the shader. Stock definition builders still keep the declaration in the `ShaderDefinition` too, then bind it into the shader through the shader-interface `Unk54` call.
- `NighteyeShader::NighteyeShader` at `0x007F5780` allocates object size `0xC0`, owns one vertex program slot, one pixel program slot, and one pass slot, then sets the base initialized byte. Its destructor at `0x007F4E50` releases exactly those three resources before base teardown.
- `RefractionShader::RefractionShader` at `0x00800B70` allocates object size `0xC4`, owns one vertex program, one pixel program, one pass slot, and an extra refcounted texture pointer at `+0xA0`. Its destructor at `0x00800110` releases those resources and also returns global `dword_B474AC` through `sub_7C1EE0` before clearing it.
- `GethitShader::GethitShader` at `0x007EC7D0` allocates object size `0xB4`, owns three vertex programs, three pixel programs, two pass slots, and one mode/state word at `+0x90`. Its destructor at `0x007EB110` releases all six programs and both pass slots before base teardown.
- `CopyShader::CopyShader` at `0x00804790` allocates object size `0xB8`, owns two vertex program slots at `+0x90`, three pixel program slots at `+0x98`, and a mode/variant field at `+0xA4`. Its program binder `0x008046C0` selects those shader programs from `+0x90/+0x98` based on `+0xA4` before the draw.
- Consequence for Moon Sugar: a native synthetic shader cannot be just a naked function pointer in the image-space list. It needs a concrete `BSImageSpaceShader`-derived object shape with program arrays, pass slots, source texture lifetime, active predicate, program load, pass construction, render method, and destructor behavior that matches the engine's refcount assumptions.

Shader-definition ownership:

- `GetShaderDefinition` at `0x007B4290` is a lazy dispatcher/cache over IDs `1..0x1B`, stored at `0x00B42EC0 + id * 4`. Image-space-relevant IDs observed here are `0x07` Blur, `0x08` HDR, `0x0B` Hit, `0x0C` Copy, `0x12` NightEye, and `0x15` Refraction.
- The dispatcher itself enforces important gates: `0x08` returns HDR only when `UseHDR` is enabled; `0x0B`, `0x15`, and `0x18` return null when `ShaderPackage < 2`; `0x1A` returns a definition only when `UsePS3Shaders` is enabled. `0x0C` Copy and `0x12` NightEye are always available after shader init.
- Shader system init at `0x007BA0F0` prewarms shader definitions `1..27`, then constructs `imageSpaceShaderList` if `ImageSpaceEffectEnabled` is enabled.
- Each stock image-space definition allocates an 8-byte `ShaderDefinition`, creates a `NiDX9ShaderDeclaration`, allocates its concrete shader object, calls vtable `+0x84`, `+0xB0`, and `+0xB4`, stores the shader in the definition with refcounting, and binds the declaration through the shader interface.
- The image-space constructors share the same declaration pattern: `CreateDX9ShaderDeclaration(dword_B43104, 2, 1)` with two elements added before program load. Copy, Hit, Refraction, and NightEye all follow this template, differing mainly in concrete object size, shader file load, constant maps, texture stages, and active predicate.
- CopyShader at `0x00804C50` is the minimal definition pattern: allocate definition, create the two-element declaration, allocate a `CopyShader` object, call vtable `+0x84`/`+0xB0`/`+0xB4`, refcount it into `ShaderDefinition::shader`, then bind the declaration via shader-interface `Unk54`.
- `NightEyeShader` at `0x007F59C0`, `RefractionShader` at `0x00800E20`, and `HitShader` at `0x007EC950` repeat the same definition pattern with object sizes `0xC0`, `0xC4`, and `0xB4` respectively. This confirms the stock image-space family does not need a special external factory for each concrete object beyond the cached `GetShaderDefinition` dispatcher.
- RefractionShader at `0x00800E20` repeats the same definition pattern with a larger concrete object. Its pass construction `0x00800900` creates two texture stages, calls the shared texture-stage initializer for each stage, attaches both stages to its pass, and applies the same broad image-space render-state disables. This is the clearest native proof that a two-input Moon Sugar image-space pass is structurally valid when the shader object owns the second texture/stage.
- This confirms a synthetic native `BSImageSpaceShader` is possible in principle but is not a small patch: it needs a valid `ShaderDefinition`, declaration, concrete vtable-compatible object, refcount/lifetime management, recreate handling, and correct constants/pass setup.

Native shader program load/cache:

- Native image-space loaders pass HLSL source paths such as `imagespace\1x\v\copy.v.hlsl` plus cache names such as `COPY000.vso`/`COPY000.pso` into `CreateVertexShader` at `0x008014E0` or `CreatePixelShader` at `0x00801C90`.
- `sub_801030` resolves the HLSL path against Oblivion's shader base path. `sub_801210` builds the compiled cache filename from `CacheOrNullString` or `FullPath`, appends `HDR.` when `UseHDR` is active, then appends the requested cache filename.
- Both loaders first try to use compiled cache data when the cache file exists and is not older than the HLSL source. If the cache is unavailable or stale, they open the HLSL source, build a D3DX macro list, use `ShaderIncludes`, and compile entry point `main`.
- `CreateVertexShader` uses the requested vertex profile directly, for example `vs_1_1` from Refraction. `CreatePixelShader` special-cases requested `ps_2_0` through `sub_7B4780(1)`, falling back to the requested profile if that helper returns null.
- On successful compile, the loaders write the compiled blob back to the `.vso` or `.pso` cache file, then call the D3D device `CreateVertexShader` or `CreatePixelShader` and wrap the result in concrete `NiD3DVertexShader` or `NiD3DPixelShader` objects derived from `NiD3DShaderProgram`. `CreateVertexShader` allocates a `0x38`-byte vertex wrapper, constructs it through `sub_780D80`, and stores the `IDirect3DVertexShader9*` at wrapper `+0x30` through vtable `+0x44`. The Hex-Rays return type is often only the base pointer.
- The standalone `CreateVertexShader` success tail is exact enough for a plugin-owned object replacement. At `0x00801B93` it calls `dword_B43104->device->CreateVertexShader(bytecode, &handle)`, allocates `0x38` bytes at `0x00801BCC`, constructs with `sub_780D80(dword_B43104)` at `0x00801BF0`, and stores the D3D handle at `0x00801C14` through vtable `+0x44`. This path does not fill all metadata fields; it returns a valid wrapper because the renderer/device and D3D handle are the fields consumed by pass binding and destruction.
- The related shader-program-creator paths `sub_7814C0`/`sub_781670` for HLSL and `sub_781B10`/`sub_781C60` for assembled/object bytecode perform similar wrapper construction, then fill metadata through base vtable slots: source/cache name at `+0x08`, profile/string field at `+0x10`, blob-size/blob-pointer fields at `+0x1C`, creator pointer at `+0x24`, and vertex-only fields through `+0x3C/+0x4C/+0x54`. Their helper `sub_783BF0` creates `IDirect3DVertexShader9` from bytecode using global device `dword_B428C0` vtable `+0x16C`; the extra arguments passed by the creator are ignored by that helper.
- The standalone TES4 `CreateVertexShader` loader is simpler than the creator-backed path. It returns a valid `NiD3DVertexShader` wrapper, but it does not set the base creator pointer or retained bytecode metadata. Stock image-space and object shader classes handle this by rerunning their loaders during class/resource setup, not by relying on the wrapper to restore itself.
- Consequence for a candidate native Moon Sugar shader: the lower-risk native-style path appears to be an HLSL source and cache-name pair following this loader model. The current plugin-owned bytecode path remains lower risk for deployment, but a native `BSImageSpaceShader` candidate would need to use the stock loader conventions or carefully mimic the resulting `NiD3DShaderProgram` wrapper ownership.

Native shader wrapper and pass binding:

- `NiD3DShaderProgram::NiD3DShaderProgram` at `0x00782C60` initializes the `NiRefObject` refcount, clears base fields through `+0x24`, then calls `sub_782BF0`. `sub_782BF0` stores the renderer at `+0x20`, `AddRef`s `renderer + 0x280` into `+0x1C`, and caches `renderer + 0x8AC` at `+0x24`.
- Pixel shader wrappers are `0x2C` bytes. `sub_780B90`, `sub_781DC0`, and `sub_781ED0` construct a `NiD3DPixelShader`, clear `+0x28`, fill the base metadata strings/profile/blob fields, and store the D3D pixel shader handle through vtable `+0x3C` (`sub_780D70`). The getter used by render code is vtable `+0x38` (`sub_780D60`), returning `+0x28`.
- Vertex shader wrappers are `0x38` bytes. `sub_780D80`, `sub_781B10`, and `sub_781C60` construct a `NiD3DVertexShader`, clear byte `+0x28` and dwords `+0x2C/+0x30/+0x34`, fill the base metadata when a creator is involved, and store the D3D vertex shader handle through vtable `+0x44` (`sub_780DF0`). The getter used by render code is vtable `+0x40` (`sub_780DE0`), returning `+0x30`. Vertex-only `+0x38/+0x3C` get/set wrapper `+0x2C`, `+0x4C` sets wrapper `+0x34`, and `+0x50/+0x54` get/set byte `+0x28`.
- Wrapper destructors preserve the same ownership split. `NiD3DPixelShader::~NiD3DPixelShader` at `0x00782E40` tears down pixel-shader state and clears/releases the `+0x28` handle path before the base destructor. `NiD3DVertexShader::~NiD3DVertexShader` at `0x00783090` mirrors that for the vertex `+0x30` handle path.
- The vertex wrapper vtable is now bounded for the slots that matter to pass mutation: `+0x28` is `sub_7830C0`, the normal scalar/vector constant setter; `+0x2C` is `sub_783180`, the indexed float-array setter for entry types `8` and `9`; `+0x40/+0x44` are the D3D vertex shader handle getter/setter; `+0x58` is `sub_783070`, which releases and nulls the live vertex shader handle through `sub_763090`; and `+0x5C` is `sub_783080`, the validity/restore thunk over `sub_77EB10`. The pixel wrapper mirrors handle ownership at `+0x38/+0x3C` and uses `+0x30` for the normal constant setter.
- `NiD3DPass::ApplyPassToRender` at `0x0075FD90` reads vertex-wrapper vtable `+0x50` before render states and passes that byte to `NiDX9RenderState::SetSoftwareVertexProcessing`. Since vtable `+0x50` returns wrapper byte `+0x28`, a Moon Sugar replacement wrapper should leave `+0x28 == 0` unless software vertex processing is deliberate.
- Vertex wrapper destruction is renderer-state aware. `NiD3DVertexShader::~NiD3DVertexShader` calls `sub_763090` when wrapper `+0x30` is live; that helper clears the current vertex shader through renderer-state vtable `+0x88`, releases the D3D handle, and stores null through vtable `+0x44`. A plugin must not raw-release or raw-swap a D3D shader handle while a native wrapper or pass still believes it owns it.
- `NiD3DShaderProgramFactory::NiD3DShaderProgramFactory` at `0x0077F500` creates two string pointer maps: vertex programs at factory `+0x18` and pixel programs at `+0x1C`. `sub_77EFA0` removes a vertex wrapper from the `+0x18` map by the wrapper key returned through base vtable `+0x04`; `sub_77EFD0` mirrors that for pixel wrappers and the `+0x1C` map. This is factory bookkeeping, not pass ownership.
- `sub_77EB10` is the vertex-program validity/restore helper reached through wrapper vtable `+0x5C` and HLSL wrapper analogues. If wrapper `+0x40` already returns a live D3D handle it succeeds. Otherwise it calls base vtable `+0x20` to read the creator pointer stored in base field `+0x18`, then calls that creator's vtable `+0x1C` to restore/recreate the handle. `sub_77EB50` is the pixel analog, using pixel handle getter `+0x38` and creator vtable `+0x20`.
- A standalone wrapper returned by TES4 `CreateVertexShader` has renderer/device ownership and a live D3D handle, but no creator pointer and no retained bytecode blob. It is safe for pass binding and normal refcount destruction, but it cannot self-restore through `sub_77EB10` after its handle is released or lost. For Moon Sugar, either rebuild the replacement wrapper when native shader resources are reloaded, or deliberately implement the heavier creator-backed wrapper path with retained bytecode and a creator pointer.
- `sub_7AEC60` is the pass pixel-program ref setter. It replaces `NiD3DPass +0x44` with a `NiD3DPixelShader*`, releases the previous ref, and increments the new wrapper ref. `sub_7AECB0` is the pass vertex-program ref setter, doing the same for `NiD3DPass +0x58` and `NiD3DVertexShader*`.
- Concrete image-space program loaders all store returned wrappers with the same ref-counted replace pattern before pass binding. Confirmed loaders are `sub_7F5350` for NightEye, `sub_800750` for Refraction, `sub_7EBC80` for Hit/double-vision, and `sub_8041B0` for Copy. Each decrements the old wrapper ref at the object slot, writes the new `CreateVertexShader` or `CreatePixelShader` result, then increments the new wrapper ref.
- The loader files and cache names are concrete: NightEye uses `base.v.hlsl`/`nighteye.p.hlsl` with `NIGHTEYE%03i`; Refraction uses `base.v.hlsl` plus `TEX2` and `refraction_P20.p.hlsl` with `REFRACT2%03i`; Hit uses three `ISHIT2%03i` variants over `base.v.hlsl`, `copy.p.hlsl`, `blur_20.p.hlsl`, and `doublevis_P20.p.hlsl`; Copy uses `copy.v.hlsl` and `copy.p.hlsl` with `COPY%03i` plus `MASK` variants.
- `CopyShader`'s binder at `0x008046C0` demonstrates the stock image-space binding model: choose wrapper slots from the concrete shader object (`+0x90/+0x94` vertex, `+0x98/+0x9C/+0xA0` pixel according to mode `+0xA4`) and copy those wrapper refs into the pass through `sub_7AECB0` and `sub_7AEC60`.
- Draw-time unwrapping is deferred to `NiD3DPass::sub_75FBA0` at `0x0075FBA0`. It reads `CurrentPass->PixelShader`, calls its vtable `+0x38` to get `IDirect3DPixelShader9*`, passes that to `renderStateG->SetPixelShader`, then applies the pass pixel constant map. It then reads `CurrentPass->VertexShader`, calls vtable `+0x40` to get `IDirect3DVertexShader9*`, passes that to `SetVertexShader`, and applies the pass vertex constant map.
- Consequence for a candidate native Moon Sugar shader: raw D3D shader handles should not be stored directly in the concrete image-space object or pass. A native-style object would likely need to own `NiD3DPixelShader`/`NiD3DVertexShader` wrappers, bind those wrappers into `NiD3DPass +0x44/+0x58`, and let `NiD3DPass::sub_75FBA0` perform the final render-state binding.
- Consequence for object vertex replacement: use the stock `CreateVertexShader` path to get a real `NiD3DVertexShader` wrapper, or exactly mirror `sub_780D80`/`sub_782BF0`/`sub_780DF0` before any pass touches it. Install the replacement wrapper with `sub_7AECB0`, restore the original wrapper with `sub_7AECB0`, and release by wrapper refcount only. Treat that wrapper as plugin-rebuildable on renderer resource events rather than factory-restorable unless the creator-backed path is implemented.

Native pass and texture-stage ownership:

- `NiD3DPass_NiD3DPass` at `0x007601E0` initializes `CurrentStage`, `StageCount`, `TexturesPerPass`, a `NiTArray<NiPointer<NiD3DTextureStage>>` sized from `dword_B28CB0`, render-state group, pass-owned pixel/vertex constant maps, shader program filename/entry/target strings, pixel/vertex shader pointers, software vertex-processing flag, renderer-owned flag, refcount, and name.
- Stock image-space pass builders do not directly `FormHeapAlloc` passes. `sub_7606A0` at `0x007606A0` checks a `NiD3DPass` out of the renderer-owned global pool `dword_B42044`, increments its refcount, and marks `RendererOwned = 1`.
- `sub_7604D0` at `0x007604D0` is the matching renderer-owned pass release. It releases pass-owned pixel and vertex constant maps, pixel and vertex shader refs, render-state group, owned shader filename/entry/target strings, texture stages, stage counters, software VP flag, then returns the pass to `dword_B42044`. If the pass is not renderer-owned, it falls back to the pass destructor.
- `sub_76CCA0` grows/shrinks a shader's pass array and releases removed `NiD3DPass` refs. `sub_76CE40` stores a pass pointer into a pass-array slot, updates object count, releases the previous pass, and increments the new pass refcount. This is the array ownership path under `NiD3DShader`.
- `sub_760010` attaches or replaces a `NiD3DTextureStage` on a `NiD3DPass`, preserving reference counts and maintaining `StageCount`, `TexturesPerPass`, and `CurrentStage`. A stage with a non-null `Texture` increments `TexturesPerPass`.
- Texture stages are pooled too. `sub_772630` at `0x00772630` checks out a `NiD3DTextureStage` from global pool `dword_B4275C`, increments its refcount at `+0x5C`, and marks it pool-owned at byte `+0x5B`. `sub_772560` releases the stage: pool-owned stages clear texture/state data and return to `dword_B4275C`; non-pool stages are freed.
- `sub_801110` is the shared image-space texture-stage initializer used by Copy, Hit, NightEye, and Refraction pass builders. It writes the texture coordinate index, sets color/alpha op and argument defaults, clears projected texture state, and sets address/filter sampler states through `sub_773100`.
- `sub_771640(stage, mode)` applies one of the stock sampler filter triplets from `0x00B420E8..0x00B4210C` to sampler states `5`, `6`, and `7`. Copy/NightEye use this immediately after `sub_801110`.
- `sub_772DF0` checks out a pooled `NiD3DRenderStateGroup` from `dword_B427A8`. `sub_772CD0` inserts or updates render-state id/value pairs inside that group. The common image-space pass builders use it to disable/normalize z, blend, alpha, and related render states before the quad draw.
- During draw, `NiD3DPass::ApplyPassToRender` at `0x0075FD90` applies the render-state group, applies each attached texture stage through `sub_772400`, and calls `sub_771790` to disable all remaining texture stages. `sub_772400` validates stage state, then calls `sub_7721E0`; `sub_7721E0` resolves the `NiTexture`/`NiRenderedTexture`, binds it to the D3D texture slot, and applies sampler/stage state.
- Consequence for a candidate native Moon Sugar shader: pass construction likely needs the same pooled `NiD3DPass`, `NiD3DTextureStage`, and `NiD3DRenderStateGroup` helpers as the stock image-space classes. A plugin object that bypasses these pools would have to reproduce their cleanup and refcount behavior or it would be vulnerable to leaks, stale stages, or device/recreate crashes.

Render-target and constant-map pass:

- `GetRenderedSurfaceParameters` at `0x007C0D10` is the IDA-identified default render-target type map for the inspected binary. Type `4` is the full-size image-space source: default render-target width/height, HDR-aware format (`0x71` when HDR is enabled, otherwise `0`), and MSAA-aware usage (`0x22` normally, `0xA2` when multisampling is active).
- Type `3` is the full-size image-space ping-pong target used by `ImageSpaceShaderList::ProcessImageSpaceShader`; it uses default render-target width/height, usage `0x26`, and format `0`.
- Type `5` is the menu-rendered texture path used by `sub_40D160`; it is separate from the normal world image-space hook.
- Types `0x10` and `0x13` are square native blur/hit temporary targets sized by `dword_B2C2B8` (`iBlurTexSize:BlurShader`) with usage `6` and format `0`.
- `NiRenderer_GetRenderedSurface` at `0x007C1B50` caches/reuses surfaces by width, height, format, usage, and depth/stencil flags. Reusing engine default targets is normal behavior, not a unique Moon Sugar allocation.
- `BlurShader_P20` constant setup at `0x007E9C10` exposes pixel constants `blendW`, `blurParams`, and `alphaAdd`, plus vertex constants `texRatio0` and `texRatio1`.
- `GethitShader` constant setup at `0x007EB250` exposes `fDoubleVis` from `0x00B2D90C..` and `fBlurParams` from `0x00B46114..`, plus `texRatio0` and `texRatio1`.
- `CopyShader` fallback constant setup at `0x00803C90` exposes `texRatio0`, `RowOne`, `RowTwo`, `blurscale`, and `blurdata`. Its render pass is used when no native image-space shader is active, so the source-to-destination copy still occurs before the Moon Sugar post-pass hook resumes.
- `CopyShader` loads `imagespace\1x\v\copy.v.hlsl` and `imagespace\1x\p\copy.p.hlsl` variants in `sub_8041B0`; `sub_804510` builds the single-stage D3D pass and disables the same broad blend/depth/write states seen in the other TES4 image-space passes.
- `SetupShaderPrograms` at `0x0077A1F0` is the draw-time bridge between `NiD3DPass` and constant maps. It calls the current pass setup, then applies the pass pixel constant map to `CurrentPass->PixelShader`, applies the pass vertex constant map to `CurrentPass->VertexShader`, and finally sets the shader declaration on pass `0`.
- `NiD3DPass::sub_75FBA0` also applies pass-owned constant maps immediately after binding the pixel and vertex shader objects through `renderStateG`. This means native constants can exist both on the shader object and on the individual pass; image-space classes observed here use the pass path.
- `NiD3DShaderConstantMap::NiD3DShaderConstantMap` at `0x009A46A0` initializes the entry array and calls `sub_9A8BD0`. `sub_9A8BD0` stores the renderer pointer, `AddRef`s the D3D device at `renderer + 0x280`, and caches renderer state from `renderer + 0x8AC`. Constant maps therefore own a real D3D device reference while alive.
- `NiD3DShaderConstantMap::AddConstant` at `0x009A8660` dispatches by `Flags & 0xF0000000`: `0x10000000` direct/static data through `sub_9A8C40`, `0x20000000` named/global lookup through `sub_9A8800`, `0x30000000` object `NiExtraData` lookup through `sub_9A8890`, and `0x40000000..0x60000000` operator/material/light style paths.
- The direct/static path `sub_9A8C40` allocates a `NiD3DShaderConstantMapEntry`, copies the key string, stores shader register and register count, and either references or owns the source data pointer. It normalizes sizes into low type bits: `0x10` bytes becomes `0x10000007`, `0x100` bytes becomes `0x10000009`, matching the image-space `float4` and array constants seen above.
- The apply loop at `0x009A8E30` iterates every active entry and routes it by the same high-nibble family. For the common `0x10000000` direct/static path it either calls the D3D device directly when the entry requests that, or calls the derived pixel/vertex shader constant setter. Failures collapse to the observed `0x80000050` style constant-write errors.
- The vertex named/global constant source is `NiD3DSCM_Vertex` vtable `+0x50`, `sub_9A56D0`. Its name table at `0x00B32590` maps 108 names, including `World`, `WorldView`, `WorldViewProj`, transpose/inverse variants, `SkinWorldView`, `BoneMatrix3`, `BoneMatrix4`, `SkinBoneMatrix3`, `SkinBoneMatrix4`, texture transforms, light/material vectors, `EyeDir`, `time`, `sin_time`, `cos_time`, and `AmbientLight`.
- For normal object shaders, `sub_9A56D0` reads renderer matrices directly: projection from `NiDX9Renderer+0x9C0`, view from `+0x980`, world from `+0x940`, and multiplies those for `WorldView`, `ViewProj`, and `WorldViewProj`. This confirms that the `sub_765480` world-matrix upload is consumed by programmable vertex shaders through constant maps, not only by fixed-function `SetTransform`.
- Bone constants are a separate family: `BoneMatrix3/4` and `SkinBoneMatrix3/4` use the skin partition/bone-array arguments passed into the constant-map apply call. They do not simply read the non-skinned world matrix. A lower-risk first Moon Sugar geometry wobble should skip skinned geometry instead of trying to perturb these arrays.
- `NiD3DPixelShader` vtable `+0x30` resolves to `sub_782E70`. It chooses the D3D call by the entry type table: type `1` writes `SetPixelShaderConstantB` through device vtable slot `0x1C4`, type `3` writes `SetPixelShaderConstantI` through slot `0x1BC`, and the normal float path writes `SetPixelShaderConstantF` through slot `0x1B4`.
- `NiD3DVertexShader` vtable `+0x28` resolves to `sub_7830C0`. It mirrors the pixel path with `SetVertexShaderConstantB/I/F` through device vtable slots `0x188`, `0x180`, and `0x178`.
- `NiD3DVertexShader` vtable `+0x2C` resolves to `sub_783180`, the indexed float-array path. It accepts entry types `8` and `9`, then calls `SetVertexShaderConstantF` for selected rows using the index list argument. This matches the bone/matrix-palette style constant path and reinforces the current Moon Sugar rule to skip skinned geometry until the skin partition and bone-index arguments are fully owned.
- Consequence for a synthetic native Moon Sugar image-space shader candidate: constants should likely be pass-owned `NiD3DShaderConstantMap` entries using the same direct/static family as Blur, Gethit, and Copy. A compact native-style layout would use pixel registers for time/intensity/tuning and vertex registers `c6/c7` for texture ratios, rather than plugin-side raw `SetPixelShaderConstantF` calls.
- Consequence for true object wobble: a rigid object sway can be done by perturbing the transform before `sub_765480` and letting Oblivion's normal constant maps propagate the new `World`/`WorldViewProj`. A non-rigid edge ripple still requires a vertex shader that actually displaces vertices; the decoded constant maps can provide stock matrices and stock time-style constants, but existing shaders will not wiggle vertices unless their shader code references such a displacement.
- A draw-time vertex-program swap is now bounded but still not first-choice. `NiD3DShader::SetupShaderPrograms` at `0x0077A1F0` calls the current pass binder, applies pass and shader constant maps, and then sets the shader declaration on pass zero. `NiD3DPass::sub_75FBA0` binds the current pass `NiD3DVertexShader` if present; otherwise it clears the vertex shader and binds either FVF or the existing `BuffData` declaration. Replacing the vertex shader after this point would have to preserve the exact declaration, constant expectations, stream layout, fixed-function fallback behavior, and pass ownership for each material.
- `sub_7AECB0` and `sub_7AEC60` are real refcounted pass shader setters, but pass objects are shared native resources. Mutating a stock pass to install a Moon Sugar vertex shader would leak across materials/passes unless every mutation is scoped and restored perfectly, and it would still need shader variants for every observed declaration family. For non-rigid object wobble, the least reckless experiment would be a whitelist-only draw hook for one positively identified declaration family, not a global pass mutation.

Full non-rigid per-vertex deformation target:

- Shader definition ID `1` at `0x007CBDC0` constructs the base `ShadowLightShader` from four declarations: a 6-element normal declaration, an 8-element declaration, a 4-element compact declaration, and an 8-element blend-related declaration. Its constructor `0x007C88B0` stores those declarations at the base shader object slots used by later object shader families. This is the IDA-identified base for Parallax/Skin constant-map behavior.
- `ShadowLightShader` vtable `+0x88` is `0x007CC280`. It constructs the base vertex/pixel constant maps. Parallax and Skin do not build their own normal constant maps; their shared vtable `+0x88` helper `0x00808DF0` ref-copies `PixelConstantMap` and `VertexConstantMap` from `GetShaderDefinition(1)->shader`, then copies the base device/renderer/render-state references.
- The base `ShadowLightShader` main vertex map includes `WorldViewProjTranspose`, `SkinWorldViewProjTranspose`, `ShadowProj`, `SkinWorldTranspose`, `WorldTranspose`, `ShadowProjData`, `ShadowProjTransform`, high-detail range, directional/point light vectors, diffuse/fog/eye/bound constants, `BoneMatrix3`, `WorldViewTranspose`, `SkinWorldViewTranspose`, and direct `Time` at register argument `0x16` backed by `flt_B44EE0`.
- Lowercase named constants such as `time`, `sin_time`, `cos_time`, `tan_time`, and `time_sin_cos_tan` are a different path. They come from the global constant-name table at `0x00B32590` and `sub_9A56D0` resolves them through object extra data named `NDL_UpdateTime`. ShadowLight's uppercase direct `Time` constant is global/static map data; it is not the same mechanism and should not be treated as a Moon Sugar intensity channel.
- `ShadowLightShader` also builds an alternate vertex map at object slot `+0x90`: `WorldViewProjTranspose` at register argument `0`, `SkinWorldViewProjTranspose` at `4`, `BoneMatrix3` starting at `0x0E`, `EyePosition` at `8`, `U Offset` at `9`, `V Offset` at `0x0A`, `Z scaler` at `0x0B`, `FogParam` at `0x0C`, and `FogColor` at `0x0D`. This map is useful evidence for how TES4 packs alternate vertex constants, but it is not proof that a Parallax draw is using those registers on the normal path.
- A Moon Sugar replacement vertex shader should consume the stock register layout for the whitelisted family and use a plugin-owned raw constant for effect intensity/tuning. Blindly squatting on spare-looking stock registers is unsafe because the base map can also write decals, shadow projection, bone matrices, and pass-specific constants depending on shader package and draw mode.
- The first lower-risk non-rigid candidate remains non-skinned Parallax pass `0`: `sub_8067F0` maps `dword_B42E90 == 0x4C` to selected pass index `0`, helper `0x008736F0`, and global pass `dword_B47620[0]`. The pass population path `0x00864A60` binds the vertex shader wrapper from the Parallax object at `this+0xDC` (`this[0x37]`, PAR2 vertex slot `0x10`) and the pixel shader wrapper from `this+0x15C` (`this[0x57]`).
- The Parallax loader narrows that target to `PAR2016.vso`: slot index `0x10` at object `+0xDC` is built from `lighting\2x\v\AD.v.hlsl` with `LIGHTS=2` and `PARALLAX`, with no `SKIN` or `PROJ_SHADOW` define. The packaged `shaderpackage019.sdp` CTAB reports target `vs_2_0` and constants `ModelViewProj c0-c3`, `LightDirection c13`, `LightPosition c16-c17`, and `EyePosition c25`. Candidate Moon Sugar scratch constants for this whitelisted variant are `c26/c27` after stock maps; avoid stock `c22` `Time`, `c23/c24` fog, `c30+` decal/extra paths, and `c42+` skin/bone ranges.
- The normal 6-element declaration for the non-skinned Parallax path is `POSITION` float3, `NORMAL` float3, `COLOR`, `TEXCOORD0` float2, `TANGENT0` float3, and `BINORMAL0` float3. This comes from `sub_76FB30`/`sub_76FC40` and materialization in `sub_770150`; that is the engine declaration order, not the final shader read order. Direct `PAR2016.vso` bytecode consumes `v0 POSITION`, `v1 TANGENT`, `v2 BINORMAL`, `v3 NORMAL`, and `v4 TEXCOORD`; the declared color channel is present in the layout but unused by this variant.
- Direct disassembly of `shaderpackage019.sdp` closes the first whitelisted shader contract. `PAR2016.vso` starts at package offset `0x8C564`, targets `vs_2_0`, transforms `v0` through `ModelViewProj c0-c3` to `oPos`, writes base UV to `oT0.xy`, derives normalized tangent-space directional light from `LightDirection c13` into `oT1.xyz`, derives tangent-space point-light direction from `LightPosition c17` into `oT2.xyz`, writes attenuation-map coordinates to `oT4.xyzw`, and writes normalized tangent-space view direction from `EyePosition c25` into `oT7.xyz`.
- The paired pass-0 pixel program is `PAR2012.pso` at package offset `0x997A0`, target `ps_2_x`. - A low-frequency Moon Sugar camera sway can be implemented at the camera setup path, then combined with the existing image-space distortion pass.
